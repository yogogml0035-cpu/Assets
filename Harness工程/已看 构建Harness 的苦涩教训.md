构建 **harness**的苦涩教训是，你手工写进系统里的每一条规则，都可能在未来变成**agent**的能力上限**。Browser-use**联合创始人**Gregor Zunic**最近发表了一篇文章《**The Bitter Lesson of Agent Harnesses**》。它的核心观点是：你替**agent** 预先写死的规则越多，越可能限制它在运行时自己长出能力。

这背后有一个变化：过去的软件世界里，作者在写代码时先把路都铺好，用户运行时只能沿着路走。但 **agent**不是这样**，agent**越强，越像一个运行时工程师——他一边执行一边读错误，一边观察一边改环境**。Browser-use**做的浏览器**agent** 可以理解成让模型替你操作浏览器。你给他一个任务，比如：
- 登录后台查员工生日；
- 填写表单；
- 在管理页面里改一个权限。



![](https://pic.aihaoji.com/user_3412cc21-72bc-d9dd-d569-1069dd8b6c83/img/20260424/5906970b-736f-f26b-127b-7b931ce2c38b/29560223-5576-416c-90b6-5fcad777abdf.webp)

A要做的事是看懂页面，找到按钮和输入框，点击输入翻页，再根据结果继续行动。这件事听起来像人操作网页，但落到工程里，要先解决一个问题：模型怎么控制 **Chrome**。

这里会遇到两个词：

1. **DOM**，可以理解成网页在浏览器里的结构树——按钮、输入框、标题列表都会挂在这棵树上；
2. **CDP**（全称**ChromeDevTools Protocol**），可以理解成 Chrome 暴露出来的底层控制通道。

平时我们打开开发者工具，看页面结构、抓网络请求、执行 JavaScript 这些能力背后，很多都能通过 **CDP** 直接调用。比如：

- 页面跳转可以用 **Page.navigate**；
- 找 **DOM**元素可以用**DOM.querySelector**；
- 执行 JavaScript 可以用 **Runtime.evaluate****。Browser-use** 最早做这件事时，走的是传统工程师很自然会走的路：先把网页复杂性包装成更友好的工具。



![](https://pic.aihaoji.com/user_3412cc21-72bc-d9dd-d569-1069dd8b6c83/img/20260424/5906970b-736f-f26b-127b-7b931ce2c38b/10c30336-e49f-41d3-8b51-c3c01fc7a292.webp)

所以他们写了几千行代码：

- **页面元素提取器**：把页面元素先抽取出来；
- **DOM索引器**：给复杂DOM做一套索引；
- **点击包装器**：把底层点击包成更好用的点击动作。

放在传统软件里，这几乎是标准做法。工程师先理解复杂系统，再把稳定接口交给上层使用。但 **Agent** 面对的是另一种问题——这些辅助函数确实让接口更整齐，同时也把模型锁定了作者预设的动作集合。

点击、输入、滚动看起来是方便函数，但在 **Agent** 这里，它们也是约束。你以为自己在帮模型，其实是在替他提前决定什么动作可用。它不只包装动作，也会包装错误：

- 系统可能崩，就写 **监控服务**；
- 浏览器可能断，就写处理器；
- 页面状态可能乱，就把异常提前兜住。

在普通软件里，这依然合理，但放到 **Agent**系统里，它会变成另一种**Harness**。比如**Chrome** 可能出现：
- Tab崩溃
- 目标断开
- 渲染进程内存溢出（OOM）
- GPU进程崩溃

每一种异常都可以写一个处理器，听上去更稳，代价是系统越来越重。因为 **Chrome** 内部状态一直在变，每个处理器都要持续跟着同步。

到这里，**传统封装**（The Old Way）的问题就清楚了：它想用手工规则同时包住动作和错误，结果**Harness**越来越厚。新做法的反转在这里：复杂性未必该被藏起来。对会写代码的**Agent** 来说，真实错误本身就是可以利用的信息。



![](https://pic.aihaoji.com/user_3412cc21-72bc-d9dd-d569-1069dd8b6c83/img/20260424/5906970b-736f-f26b-127b-7b931ce2c38b/6a311931-08ce-44a4-bff7-daa8986a7d89.webp)

那他们现在怎么做？答案是把 **中间层削薄**，保留一个很小的**harness**，再把原始**CDP**直接交给模型**。Rowtheruse**现在的**harness** 核心只有4个文件：  
- `runpy.py`  
- `helpers.py`  
- `daemon.py`  
- `skill.md`  

`runpy.py` 负责运行普通 **Python**脚本，`helpers.py` 放很薄的**CDP**辅助函数，`daemon.py` 维持**CDP**网络连接，`skill.md` 告诉**agent** 怎么用。整套大约600行，`helpers.py` 只有192行。  

它不是一个大框架，更像一个可改写的工作台。**agent**写**Python**调辅助函数，辅助函数直接让**Chrome**收到命令后执行对应动作**。关键在于最后一句**：**Chrome**以上的部分都可以被**agent** 改写。工具缺了，它可以补；流程断了，它可以读错误后重试。  

这样做的第一个效果是：模型不用学习你私有封装的方言，它直接面对 **Chrome** 原本的接口。这些命令听起来底层，但模型在训练语料里见过大量真实接口。  

第二个效果是复杂页面更容易处理。比如 **cross-origin iframe**，过去你可能要发明一套页面框架抽象，现在可以直接连接目标页面。再比如**Shadow DOM**，模型看到的是**shadowRoot.querySelectorAll**，这正是代码世界里反复出现的写法。  

第三个效果是稳定性来自运行时修复。页面死了，它可以读报错；目标挂错了，它可以重新连接；**Chrome** 卡住了，它可以重试；工具缺了，它可以补工具。  

旧世界的软件更像作者时刻决定一切，新世界的 **agent** 运行时决定越来越多。接受这个前提后，设计目标就会变化：重点不再是提前写完所有工具，重点变成让模型在一个最小环境里自己补齐缺口。  

这就是 **heal loop**：工具缺了，不要等作者回家补，让**agent** 自己补。



![](https://pic.aihaoji.com/user_3412cc21-72bc-d9dd-d569-1069dd8b6c83/img/20260424/5906970b-736f-f26b-127b-7b931ce2c38b/5de363b8-01a0-4031-8eaa-dd34c8f09138.webp)

他会先搜索 **helpers.py**，发现没有这个函数，然后把缺的那一小段写进去，再重新运行。这不是从零造系统，更像你在一个代码库里发现少了个导入语句，顺手补上。

所以模型补的往往不是大工程，它补的是眼前刚好缺的那一个洞。一个例子就能说明问题：他们忘了写 **upload_agent**遇到文件输入框后，自己去搜**helpers.py**，再补上**DoSetFileInputFiles**。后来上传**12MB文件** 时，又撞上网络传输上限，它读到报错，直接切成分块上传。

这说明他不只会补工具，还会顺着真实错误继续修工具。完整任务里，他能把TTO员工生日写进 **Google Calendar**，也能处理ISER后台里大量**iframe**和**blade**。这里真正要追问的是：**能力到底长在哪里**？



![](https://pic.aihaoji.com/user_3412cc21-72bc-d9dd-d569-1069dd8b6c83/img/20260424/5906970b-736f-f26b-127b-7b931ce2c38b/39e575d1-37c1-4a57-9200-ba0ea170b359.webp)

过去我们习惯觉得能力长在作者写好的代码里，代码越全，系统越强。答案是，在 **Agent 时代**，很多关键能力开始长在模型和环境的运行时互动里。你不给它预做完，它反而更容易在现场长出来。你替它包太多，它反而被锁死在你预设的动作集合里。

所以 **Harness**的苦涩教训，最后就是一句话：少写死规则，多留下运行时生长的空间。这里是**慢学AI**，我们下期再见。
