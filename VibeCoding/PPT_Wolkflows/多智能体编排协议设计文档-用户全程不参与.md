## 0. 协议摘要

本协议定义一套用于长时、多阶段、可复盘任务的多智能体协作机制。

系统由四类角色组成：

```text
User            初始计划确认者
Orchestrator    机械调度器 / 状态机 / 日志员
Planner Agent   唯一计划权威
Dev Agent       开发执行者
Tester Agent    只读事实验证者
```

本协议的核心变化是：

```text
主智能体不再是“大脑”。
主智能体不生成计划，不判断计划，不写业务代码，不判断失败根因。
```

真正的计划权属于 `Planner Agent`。Planner 负责：

```text
- 生成初始计划
- 在信息不足时写保守假设
- 生成初始计划确认包
- 用户确认后生成冻结基线
- 决定每一步 next_action
- 修改执行路径
- 处理 BLOCKED 和三轮 FAIL
- 决定局部搁置、临时隔离、最终完成或最终硬失败
```

主智能体只负责读取结构化字段，并按固定路由执行。

## 1. 协议目标与非目标

### 1.1 协议目标

本协议要解决的问题不是“让一个智能体变得更聪明”，而是通过协议让多个智能体在长期任务中保持边界清晰、上下文可控、失败可追溯。

目标包括：

```text
1. 计划权单一化：只有 Planner 能生成或修改计划。
2. 调度机械化：Orchestrator 只读结构化字段，不做语义判断。
3. 事实结构化：Dev/Tester 只输出事实、证据、尝试步骤和状态。
4. 文件权威化：计划、执行状态、冻结基线、临时物、日志分别由不同文件承担。
5. 失败有出口：允许局部搁置和最终硬失败，禁止低质量通过。
6. 假设可冻结：首次规划的信息不足由保守假设处理，用户确认后进入冻结基线。
7. 最终完成可验证：项目完成必须通过 Final Completion Gate。
```

### 1.2 明确非目标

本协议不追求以下目标：

```text
- 不追求主智能体自由发挥。
- 不追求执行阶段持续向用户澄清。
- 不追求失败后降低标准继续推进。
- 不追求让测试智能体修代码。
- 不追求让开发智能体判断计划问题。
- 不追求把所有状态塞进一个 plan.md。
```

### 1.3 协议关键词

本文使用以下语义：

| 关键词 | 含义 |
|---|---|
| MUST / 必须 | 协议强制要求，违反即视为流程错误 |
| MUST NOT / 禁止 | 明确禁止，违反即视为越权 |
| SHOULD / 应该 | 强烈建议，除非有明确理由不得偏离 |
| MAY / 可以 | 可选能力，不构成强制流程 |

## 2. 不可变协议规则

以下规则是整个系统的边界条件。

### 2.1 计划权规则

```text
R-PLAN-01  只有 Planner Agent 可以生成 doc/plan.md。
R-PLAN-02  只有 Planner Agent 可以修改 doc/plan.md。
R-PLAN-03  只有 Planner Agent 可以决定 next_action。
R-PLAN-04  只有 Planner Agent 可以判断是否 PATCH_PLAN / REGENERATE_EXECUTION_PATH / SUSPEND_TASK / TERMINAL_FAILURE / PROJECT_COMPLETE。
R-PLAN-05  Orchestrator 不得判断“下一个任务是什么”。
R-PLAN-06  Dev/Tester 不得判断“这是不是计划问题”。
```

### 2.2 主智能体限制规则

```text
R-ORCH-01  Orchestrator 不得生成计划。
R-ORCH-02  Orchestrator 不得修改任务顺序、任务粒度、验收标准或依赖关系。
R-ORCH-03  Orchestrator 不得写业务代码、架构代码、公共组件或测试代码。
R-ORCH-04  Orchestrator 不得自行解释用户需求。
R-ORCH-05  Orchestrator 不得自行总结 Dev/Tester 的失败根因。
R-ORCH-06  Orchestrator 只允许读取结构化字段并执行固定路由。
```

### 2.3 用户交互规则

```text
R-USER-01  首次规划阶段，Planner 禁止 ASK_USER。
R-USER-02  首次规划信息不足时，Planner 必须做保守假设。
R-USER-03  初始计划必须以 Initial Approval Package 形式提交给用户确认。
R-USER-04  用户确认初始计划 = 确认其中所有保守假设。
R-USER-05  用户确认后，所有保守假设进入 doc/confirmed-baseline.md。
R-USER-06  执行阶段永远禁止 ASK_USER。
R-USER-07  后续不能要求用户补充需求、确认重规划或修改基线。
```

### 2.4 冻结基线规则

```text
R-BASE-01  doc/confirmed-baseline.md 是确认后的契约。
R-BASE-02  确认后，任何角色不得自动修改冻结基线。
R-BASE-03  Planner 可以自动修改执行路径，但不能修改已确认需求、验收标准、技术约束、交付范围和已确认假设。
R-BASE-04  如无法在冻结基线内继续，Planner 只能尝试替代执行路径、搁置任务或宣告硬失败。
R-BASE-05  禁止为了继续执行而降低验收标准。
```

### 2.5 失败规则

```text
R-FAIL-01  Tester 输出 FAIL 时，默认先视为实现问题。
R-FAIL-02  普通 FAIL 最多进行 3 轮 Dev/Test 修正。
R-FAIL-03  第 3 轮后仍 FAIL，Orchestrator 必须把完整事实证据包交给 Planner。
R-FAIL-04  Dev/Tester 输出 BLOCKED 时，Orchestrator 必须立即交给 Planner。
R-FAIL-05  禁止 LOW_QUALITY_PASS。
R-FAIL-06  禁止“强制通过”。
R-FAIL-07  无法满足冻结基线时，最终结果只能是局部搁置或 TERMINAL_FAILURE。
```

## 3. 权威模型

本协议采用“权威分离”模型。每个关键问题必须有且只能有一个权威来源。

| 问题 | 权威角色 | 权威文件 / 数据 |
|---|---|---|
| 要做什么 | Planner | `doc/plan.md` |
| 为什么这样做 | Planner | `doc/plan.md` / `Plan Decision Packet` |
| 已确认需求是什么 | Planner 生成，用户确认后冻结 | `doc/confirmed-baseline.md` |
| 当前执行到哪一步 | Orchestrator | `doc/execution-state.md` |
| 哪个 Agent 执行过 | Orchestrator | `doc/execution-state.md` / `logs/orchestrator-log.md` |
| 某次开发做了什么 | Dev | `Agent Result Packet` |
| 某次测试观察到什么 | Tester | `Agent Result Packet` / `Final Test Result Packet` |
| 是否继续、搁置、失败、完成 | Planner | `Plan Decision Packet` |
| 临时隔离物是否允许存在 | Planner | `doc/temporary-artifacts.md` |
| 临时隔离物实际状态 | Orchestrator | `doc/execution-state.md` |
| 临时隔离物是否污染交付 | Tester 提供事实，Planner 判定 | `Final Test Result Packet` / `Plan Decision Packet` |

### 3.1 单一权威原则

同一类信息不得由多个文件同时维护。

错误示例：

```text
plan.md 同时记录任务状态、Agent ID、修正轮次、计划结构。
```

正确做法：

```text
plan.md               只记录计划结构和计划历史。
execution-state.md    只记录执行状态和 Agent 运行信息。
confirmed-baseline.md 只记录用户已确认、不可自动修改的基线。
```

## 4. 角色契约

### 4.1 User：初始计划确认者

User 的参与点只有一个：初始计划确认阶段。

User 可以：

```text
- 阅读 Initial Approval Package。
- 确认完整计划。
- 在确认前主动指出某条假设、计划或范围不对。
- 要求 Planner 基于主动指出的问题重新生成初始计划确认包。
```

User 不参与：

```text
- 执行阶段的任务级澄清。
- 三轮 FAIL 后的重规划确认。
- BLOCKED 后的需求补充。
- 最终硬失败前的临时询问。
```

注意：

```text
Planner 禁止主动问用户。
但用户在初始确认阶段可以主动改写计划或假设。
这不是 ASK_USER，而是 Initial Approval Gate 的用户审查行为。
```

### 4.2 Orchestrator：机械调度器

Orchestrator 的定位：

```yaml
identity: execution_bus / state_machine / logger
```

允许：

```text
- 创建 doc/ 与 logs/ 目录。
- 创建空白协议文件。
- 调用 Planner / Dev / Tester。
- 读取 Plan Decision Packet 的固定字段。
- 读取 Agent Result Packet 的固定字段。
- 套用不可变 Prompt Template。
- 记录 Agent ID、状态、修正轮次和日志。
- 将完整事实证据包原样转交 Planner。
```

禁止：

```text
- 生成计划。
- 修改计划。
- 自己决定下一个任务。
- 自己判断任务失败原因。
- 自己判断是否需要重规划。
- 自己总结用户需求。
- 自己写业务/架构代码。
- 自己修改验收标准。
- 自己宣告项目完成或失败。
```

Orchestrator 只能读取这些字段：

```yaml
from_plan_decision_packet:
  - packet_id
  - decision_type
  - plan_version
  - next_action.target_agent
  - next_action.action
  - next_action.task_id
  - next_action.task_payload

from_agent_result_packet:
  - packet_id
  - agent_role
  - agent_id
  - task_id
  - status

from_final_test_result_packet:
  - packet_id
  - agent_role
  - status
```

其他字段只能用于归档、转交或日志记录，不能由 Orchestrator 解释。

### 4.3 Planner Agent：唯一计划权威

Planner 的定位：

```yaml
identity: planning_authority
```

允许：

```text
- 读取需求文档、开发规范、经验库。
- 读取执行状态。
- 浏览项目目录结构。
- 读取已有代码以理解现实约束。
- 写入和修改 doc/plan.md。
- 生成 doc/confirmed-baseline.md。
- 维护 doc/temporary-artifacts.md 的计划层定义。
- 输出 Plan Decision Packet。
- 决定 next_action。
- 处理三轮 FAIL 和 BLOCKED。
- 创建临时隔离任务。
- 发起 Final Completion Gate。
- 宣告 PROJECT_COMPLETE 或 TERMINAL_FAILURE。
```

禁止：

```text
- 写业务代码。
- 修改测试代码。
- 替 Dev Agent 修 bug。
- 向用户 ASK_USER。
- 自动修改 confirmed-baseline.md。
- 降低验收标准。
- 将临时物当成正式交付。
- 输出 LOW_QUALITY_PASS。
```

Planner 必须在每次计划变更时声明：

```yaml
baseline_impact: none / possible / required
baseline_preservation_statement:
  - 本次调整是否改变已确认需求
  - 本次调整是否改变验收标准
  - 本次调整是否改变技术约束
  - 本次调整是否改变交付范围
  - 本次调整是否改变已确认假设
```

由于执行阶段禁止 ASK_USER：

```text
baseline_impact: none
→ 可以继续执行。

baseline_impact: possible / required
→ 不得修改基线；必须寻找不改基线的替代路径。
→ 替代路径耗尽后，只能 SUSPEND_TASK 或 TERMINAL_FAILURE。
```

### 4.4 Dev Agent：开发执行者

Dev 的定位：

```yaml
identity: implementation_executor
```

允许：

```text
- 读取 task_payload 指定的上下文文件。
- 读取相关代码。
- 修改授权范围内的业务代码。
- 运行自测、构建、lint 或必要命令。
- 创建 Planner 允许的临时隔离物。
- 输出 Agent Result Packet。
```

禁止：

```text
- 修改 doc/plan.md。
- 修改 doc/confirmed-baseline.md。
- 修改 doc/temporary-artifacts.md 的计划定义。
- 判断是否需要重规划。
- 判断失败是否属于计划问题。
- 建议用户补充需求。
- 降低验收标准。
- 将临时物标记为最终交付。
```

Dev 的合法状态：

```yaml
status:
  - DONE
  - BLOCKED
```

### 4.5 Tester Agent：只读事实验证者

Tester 的定位：

```yaml
identity: readonly_verifier
```

允许：

```text
- 读取 task_payload 指定的上下文文件。
- 读取待测代码和相关配置。
- 运行测试命令、构建命令、截图命令或检查脚本。
- 输出 Agent Result Packet。
- 最终验收时输出 Final Test Result Packet。
```

禁止：

```text
- 修改任何代码。
- 修改任何计划文件。
- 替 Dev 修复问题。
- 判断是否需要重规划。
- 宣告项目完成或失败。
- 建议降低验收标准。
- 向用户提问。
```

Tester 的合法状态：

```yaml
status:
  - PASS
  - FAIL
  - BLOCKED
```

## 5. 文件账本体系

### 5.1 文件总览

```text
doc/
  requirements.md              原始需求
  dev-spec.md                  开发规范
  plan.md                      Planner 维护的计划权威
  confirmed-baseline.md        用户确认后的冻结基线
  execution-state.md           Orchestrator 维护的执行状态权威
  temporary-artifacts.md       Planner 维护的临时隔离物计划定义
  user-clarifications.md       初始确认阶段用户主动修改的原始记录
  requirements-addendum.md     Planner 对用户主动修改的正式解释
  lessons-learned.md           跨任务经验库
logs/
  orchestrator-log.md          调度日志
  packets/                     可选：结构化包归档目录
```

### 5.2 `doc/plan.md`

维护者：Planner。

用途：记录计划，不记录执行状态。

必须包含：

```yaml
plan:
  plan_version: v1
  project_goal: ...
  planning_scope: ...
  task_graph:
    - task_id: T-001
      title: ...
      goal: ...
      dependencies: []
      acceptance_criteria:
        - AC-001
      expected_outputs:
        - ...
      authorized_target_files:
        - ...
      status_design_note: 不记录执行状态，状态见 execution-state.md

  assumption_register:
    - assumption_id: A-001
      assumption: ...
      why_needed: ...
      conservative_reason: ...
      affected_tasks:
        - T-001
      baseline_destination: confirmed-baseline.md

  replan_history:
    - plan_version: v2
      reason: ...
      baseline_impact: none
      changed_execution_path:
        - ...
```

禁止在 `plan.md` 中记录：

```text
- Dev Agent ID
- Tester Agent ID
- 当前执行轮次
- 第几次 FAIL
- 当前任务是否正在进行
- 临时物实际创建路径
```

这些内容属于 `execution-state.md`。

### 5.3 `doc/confirmed-baseline.md`

维护者：Planner 生成；用户确认后冻结。

用途：记录系统不可自动越过的契约。

必须包含：

```yaml
confirmed_baseline:
  baseline_version: v1
  source_plan_version: v1
  confirmed_requirements:
    - id: R-001
      content: ...
      source: doc/requirements.md

  confirmed_acceptance_criteria:
    - id: AC-001
      content: ...
      source_task: T-001

  confirmed_constraints:
    - id: C-001
      content: ...

  confirmed_delivery_scope:
    - id: DS-001
      content: ...

  confirmed_assumptions:
    - id: A-001
      content: ...
      conservative_reason: ...
      source: doc/plan.md#A-001

  non_negotiables:
    - 禁止低质量通过
    - 禁止自动降低验收标准
    - 禁止自动修改冻结基线
    - 禁止以临时隔离物冒充最终交付
    - 禁止执行阶段 ASK_USER
```

确认后：

```text
- 不得自动修改。
- 不得被 Planner “悄悄更新”。
- 不得因三轮 FAIL 而降低。
- 不得因临时隔离物存在而视为满足。
```

### 5.4 `doc/execution-state.md`

维护者：Orchestrator。

用途：记录执行状态，不记录计划设计。

必须包含：

```yaml
execution_state:
  current_plan_version: v1
  current_baseline_version: v1

  tasks:
    - task_id: T-001
      execution_status: pending / in_progress / dev_done / test_passed / test_failed / blocked / suspended / completed
      dev_agent_id: ...
      tester_agent_id: ...
      correction_round: 0
      last_packet_refs:
        - ARP-0001

  suspended_tasks:
    - task_id: T-003
      suspended_at: ...
      planner_decision_packet_ref: PDP-0017
      current_reassessment_status: pending / reassessed / terminal

  temporary_artifact_execution_state:
    - temp_artifact_id: TEMP-001
      actual_files:
        - src/mocks/temp-user-api.ts
      created_by_agent_id: ...
      associated_task_id: T-004
      current_status: planned / active / replaced / removed / unresolved
```

Orchestrator 可以更新：

```text
- Agent ID
- 状态字段
- 修正轮次
- packet 引用
- 临时物实际文件路径
```

Orchestrator 不得更新：

```text
- 任务定义
- 任务顺序
- 验收标准
- 依赖关系
- 交付范围
```

### 5.5 `doc/temporary-artifacts.md`

维护者：Planner。

用途：定义允许存在的临时隔离物。

```yaml
temporary_artifact_register:
  - temp_artifact_id: TEMP-001
    created_for_suspended_task: T-003
    created_by_plan_version: v3
    purpose: ...
    allowed_usage:
      - 仅用于 T-004 的结构开发
      - 仅用于局部验证
    forbidden_usage:
      - 不得进入最终交付
      - 不得作为 confirmed-baseline 的满足证据
    replacement_condition: ...
    must_remove_before_final: true
    planned_status: planned / active / replaced / removed / unresolved
```

### 5.6 `doc/user-clarifications.md` 与 `doc/requirements-addendum.md`

这两个文件只用于初始计划确认阶段。

`doc/user-clarifications.md`：

```text
维护者：Orchestrator
内容：用户在初始确认阶段主动指出的问题、改写或拒绝理由。
要求：原样记录，不总结、不润色、不解释。
```

`doc/requirements-addendum.md`：

```text
维护者：Planner
内容：Planner 根据用户主动修改生成的正式需求补充。
要求：必须进入重新生成的 Initial Approval Package。
```

执行阶段不得新增用户澄清。

### 5.7 `doc/lessons-learned.md`

经验库保留，但不作为主智能体的自由总结区域。

推荐机制：

```text
- Dev/Tester 在 Agent Result Packet 中提出 lessons_candidates。
- Orchestrator 可以原样归档 lessons_candidates。
- Planner 可在生成或调整计划时读取经验库。
- 需要整理经验库时，应由 Planner 或专门整理流程完成。
```

## 6. 协议生命周期

整个系统分为六个阶段。

```text
BOOTSTRAP
  ↓
INITIAL_PLANNING
  ↓
INITIAL_APPROVAL
  ↓
BASELINE_FREEZE
  ↓
EXECUTION_LOOP
  ↓
FINAL_COMPLETION_GATE
  ↓
PROJECT_COMPLETE / TERMINAL_FAILURE
```

### 6.1 BOOTSTRAP

执行者：Orchestrator。

Orchestrator 只做运行环境级初始化：

```text
- 创建 doc/ 和 logs/。
- 创建空白协议文件。
- 创建 orchestrator-log.md。
- 调用 Planner 进入 INITIAL_PLANNING。
```

禁止：

```text
- 写计划。
- 拆任务。
- 写代码。
- 搭建业务框架。
- 选择技术路线。
```

### 6.2 INITIAL_PLANNING

执行者：Planner。

Planner 读取：

```text
- doc/requirements.md
- doc/dev-spec.md
- doc/lessons-learned.md
- 项目目录结构
- 已有代码或配置
```

Planner 必须：

```text
- 生成 doc/plan.md。
- 在信息不足时写保守假设。
- 生成 Assumption Register。
- 生成 Frozen Baseline Preview。
- 输出 Initial Approval Package。
```

Planner 禁止：

```text
- ASK_USER。
- 要求用户补充需求。
- 跳过不确定性。
- 将不确定性隐藏在计划中。
```

### 6.3 INITIAL_APPROVAL

执行者：Orchestrator + User。

Orchestrator 必须原样展示 Initial Approval Package：

```text
- 完整 doc/plan.md。
- Assumption Register。
- Frozen Baseline Preview。
- 冻结说明。
```

确认说明必须明确：

```text
确认此计划 = 确认所有保守假设。
确认后这些假设会进入 doc/confirmed-baseline.md。
确认后 Planner 不得自动修改这些假设。
执行阶段不会再 ASK_USER。
如果不同意某条假设，应在确认前主动指出并改写。
```

如果用户确认：进入 BASELINE_FREEZE。

如果用户主动指出问题：

```text
1. Orchestrator 原样写入 doc/user-clarifications.md。
2. Planner 写入或更新 doc/requirements-addendum.md。
3. Planner 重新生成 doc/plan.md 和 Initial Approval Package。
4. 回到 INITIAL_APPROVAL。
```

### 6.4 BASELINE_FREEZE

执行者：Planner。

Planner 根据用户确认的 Initial Approval Package 生成 `doc/confirmed-baseline.md`。

冻结内容包括：

```text
- 已确认需求
- 已确认验收标准
- 已确认技术约束
- 已确认交付范围
- 所有保守假设
- 不可谈判项
```

完成后进入 EXECUTION_LOOP。

### 6.5 EXECUTION_LOOP

执行者：Planner + Orchestrator + Dev + Tester。

循环规则：

```text
1. Planner 输出 Plan Decision Packet。
2. Orchestrator 读取 next_action。
3. Orchestrator 套不可变模板调用 Dev 或 Tester。
4. Dev/Tester 输出 Agent Result Packet。
5. Orchestrator 读取 status 并按路由表处理。
6. PASS / BLOCKED / 三轮 FAIL 后回到 Planner。
```

### 6.6 FINAL_COMPLETION_GATE

执行者：Planner 发起，Tester 验证，Planner 宣告。

触发条件：

```text
- Planner 判断没有更多可执行 next_action。
- 所有任务状态为 completed 或 suspended。
- 所有 temporary artifacts 状态已知。
- Planner 返回 next_action: start_final_verification。
```

Tester 输出 Final Test Result Packet。

Planner 根据事实宣告：

```text
PROJECT_COMPLETE 或 TERMINAL_FAILURE。
```

Orchestrator 只归档结果。

## 8. 状态机与路由算法

### 8.1 任务执行状态

`execution-state.md` 中的任务状态建议使用：

```yaml
task_execution_status:
  - pending
  - in_progress
  - dev_done
  - test_passed
  - test_failed
  - blocked
  - suspended
  - completed
```

### 8.2 Orchestrator 路由表

| 输入来源 | 状态 / 决策 | Orchestrator 动作 |
|---|---|---|
| Planner | CONTINUE_CURRENT_PLAN | 执行 `next_action` |
| Planner | PATCH_PLAN | 记录 plan_version，执行 `next_action` |
| Planner | REGENERATE_EXECUTION_PATH | 记录 replan history，执行 `next_action` |
| Planner | SUSPEND_TASK | 更新 execution-state，执行 Planner 指定的其他任务 |
| Planner | PROJECT_COMPLETE | 归档项目，生成完成报告 |
| Planner | TERMINAL_FAILURE | 停止项目，生成失败报告 |
| Dev | DONE | 记录状态，创建或 resume Tester |
| Dev | BLOCKED | 将 Agent Result Packet 原样转交 Planner |
| Tester | PASS | 记录状态，通知 Planner 请求 next_action |
| Tester | FAIL 且修正轮次 < 3 | resume 同一 Dev，传递完整事实证据包 |
| Tester | FAIL 且修正轮次 = 3 | 将三轮事实证据包交给 Planner |
| Tester | BLOCKED | 将 Agent Result Packet 原样交给 Planner |
| Tester Final | PASS / FAIL / BLOCKED | 将 Final Test Result Packet 原样交给 Planner |

### 8.3 Orchestrator 伪代码

```pseudo
bootstrap()
call Planner for INITIAL_PLANNING
show Initial Approval Package to User

if User confirms:
    call Planner to generate confirmed-baseline.md
else if User proactively modifies assumptions/plan:
    write raw user input to doc/user-clarifications.md
    call Planner to regenerate Initial Approval Package

while true:
    pdp = call Planner or receive Planner decision

    if pdp.decision_type == PROJECT_COMPLETE:
        archive_project()
        break

    if pdp.decision_type == TERMINAL_FAILURE:
        write_final_failure_report()
        break

    action = pdp.next_action

    if action.target_agent == dev-agent:
        result = run_or_resume_dev_with_immutable_template(action.task_payload)
        record_agent_result(result)

        if result.status == DONE:
            tester_result = run_or_resume_tester(action.task_payload)
            handle_tester_result(tester_result)

        if result.status == BLOCKED:
            send_packet_to_planner(result)

    if action.target_agent == tester-agent:
        tester_result = run_or_resume_tester(action.task_payload)
        handle_tester_result(tester_result)

function handle_tester_result(result):
    record_agent_result(result)

    if result.status == PASS:
        notify_planner_for_next_action(result)

    if result.status == FAIL:
        if correction_round < 3:
            correction_round += 1
            resume_same_dev_with_full_test_packet(result)
        else:
            send_three_rounds_evidence_to_planner()

    if result.status == BLOCKED:
        send_packet_to_planner(result)
```

Orchestrator 伪代码中没有任何语义判断分支。

## 9. 初始计划确认协议

### 9.1 Planner 如何处理信息不足

首次规划时，Planner 不问用户。

如果需求缺失，Planner 必须：

```text
1. 做保守假设。
2. 写明为什么需要这个假设。
3. 写明为什么这个假设是保守的。
4. 标记受影响任务。
5. 写入 Assumption Register。
6. 在 Initial Approval Package 中显式展示。
```

示例：

```yaml
assumption_register:
  - assumption_id: A-004
    assumption: 使用现有项目技术栈，不引入新框架。
    why_needed: 需求未指定技术栈迁移或新增框架。
    conservative_reason: 复用现有技术栈能减少不确定性和破坏性变更。
    affected_tasks:
      - T-001
      - T-002
    will_enter_confirmed_baseline: true
```

### 9.2 确认即冻结

用户确认初始计划后：

```text
- 所有 assumption_register 条目进入 confirmed-baseline.md。
- Planner 后续不得自动修改这些假设。
- 执行阶段不得请求用户重新确认。
- 若假设与现实冲突，Planner 只能寻找不改基线的路径。
- 路径耗尽时，可以搁置或硬失败。
```

### 9.3 用户主动修改时的处理

如果用户在初始确认阶段主动指出问题：

```text
1. Orchestrator 原样记录用户文本。
2. Planner 读取用户原文。
3. Planner 生成 requirements-addendum.md。
4. Planner 重新生成 plan.md。
5. Planner 重新生成 Initial Approval Package。
6. 用户重新确认。
```

Orchestrator 不得总结用户意思。

## 10. 执行循环协议

### 10.1 Planner 指定下一步

每一步都由 Planner 的 `next_action` 决定。

```yaml
next_action:
  target_agent: dev-agent
  action: start_new_task
  task_id: T-004
  task_payload:
    task_title: ...
    task_goal: ...
    acceptance_criteria:
      - ...
    required_context_files:
      - doc/plan.md
      - doc/confirmed-baseline.md
      - doc/dev-spec.md
    target_files:
      - src/...
    constraints:
      - ...
```

Orchestrator 不扫描任务列表决定下一步。

### 10.2 不可变 Prompt Template

Orchestrator 不得自由组织 prompt，只能把 Planner 的 `task_payload` 套进固定模板。

Dev 模板：

```text
你是 dev-agent。

请严格执行下面 task_payload 中的任务。

禁止：
- 修改 task_payload 未授权的范围。
- 修改 doc/plan.md。
- 修改 doc/confirmed-baseline.md。
- 判断是否需要重规划。
- 向用户提问。
- 降低验收标准。

完成后必须输出 Agent Result Packet。
如果无法继续，只能输出 status: BLOCKED，并提供事实证据包。

task_payload:
{Planner 提供的原始 task_payload}
```

Tester 模板：

```text
你是 tester-agent。

请严格验证下面 task_payload 对应任务的完成情况。

禁止：
- 修改任何代码文件。
- 修改 doc/plan.md。
- 判断是否需要重规划。
- 建议降低验收标准。
- 向用户提问。
- 宣告项目完成或失败。

完成后必须输出 Agent Result Packet。
如通过，输出 status: PASS。
如未通过，输出 status: FAIL，并提供事实证据。
如无法验证，输出 status: BLOCKED，并提供事实证据。

task_payload:
{Planner 提供的原始 task_payload}
```

### 10.3 Dev/Test 修正规则

普通 `FAIL` 修正逻辑：

```text
第 1 次 FAIL → resume 原 Dev Agent 修正。
第 2 次 FAIL → resume 原 Dev Agent 修正。
第 3 次 FAIL → 不再原地修正，交给 Planner。
```

为什么是 3 轮：

```text
- 第一轮可能是普通遗漏。
- 第二轮可能是反馈理解偏差。
- 第三轮后仍失败，继续原地修正容易进入局部循环。
- 此时不由 Orchestrator 判断根因，而是把事实交给 Planner。
```

### 10.4 BLOCKED 处理

Dev/Tester 可以输出 `BLOCKED`，但不能解释成“计划问题”。

合法 BLOCKED 包示例：

```yaml
status: BLOCKED
observed_facts:
  - 目标文件不存在，且 task_payload 未授权创建该目录。
attempted_steps:
  - 检查 src/pages 目录。
  - 检查路由配置。
evidence:
  - evidence_id: E-001
    type: command_output
    value: "ls: cannot access src/pages: No such file or directory"
uncertainties:
  - 不确定是否应先创建基础路由结构。
```

Orchestrator 动作：

```text
立即转交 Planner。
不进入三轮修正。
不判断这是不是计划问题。
```

## 11. 失败、搁置与路径耗尽证明

### 11.1 Planner 可用决策

Planner 收到 BLOCKED 或三轮 FAIL 后，可以返回：

```yaml
decision_type:
  - CONTINUE_CURRENT_PLAN
  - PATCH_PLAN
  - REGENERATE_EXECUTION_PATH
  - SUSPEND_TASK
  - TERMINAL_FAILURE
```

Planner 不能返回：

```text
- ASK_USER
- LOW_QUALITY_PASS
- MODIFY_BASELINE
```

### 11.2 SUSPEND_TASK 条件

Planner 只有满足以下条件时才能局部搁置：

```text
1. 当前任务已 BLOCKED，或三轮 FAIL 后交给 Planner。
2. Planner 至少尝试过一种不改变 confirmed-baseline 的替代执行路径。
3. 当前任务暂时无法继续。
4. 仍存在其他不依赖该任务的 pending 任务。
5. 搁置不会被伪装成完成。
```

### 11.3 TERMINAL_FAILURE 条件

Planner 只有满足以下条件时才能最终硬失败：

```text
1. 所有 pending 任务都完成或被搁置。
2. 所有 suspended 任务都至少重新评估过一次。
3. Planner 无法找到任何不改变 confirmed-baseline 的执行路径。
4. 继续执行必然违反已确认需求、验收标准、技术约束、交付范围或确认假设。
```

### 11.4 路径耗尽证明

`SUSPEND_TASK` 和 `TERMINAL_FAILURE` 必须附带路径耗尽证明。

```yaml
exhaustion_evidence:
  baseline_items_involved:
    - R-001
    - AC-003

  attempted_execution_paths:
    - path_id: PATH-001
      description: ...
      result: failed / blocked
      evidence_ref: ARP-0003#E-001

    - path_id: PATH-002
      description: ...
      result: failed / blocked
      evidence_ref: ARP-0004#E-002

  why_no_baseline_preserving_path_remains:
    - ...

  remaining_unblocked_tasks:
    - T-005
    - T-006
```

Orchestrator 不判断证明是否充分，只记录并执行 Planner 决策。

## 12. 搁置影响分析与临时隔离物协议

### 12.1 搁置影响分析

当 Planner 返回 `SUSPEND_TASK` 时，必须输出：

```yaml
suspension_impact_analysis:
  suspended_task_id: T-003
  directly_blocked_tasks:
    - T-004
  indirectly_blocked_tasks:
    - T-006
  still_executable_tasks:
    - T-005
  continuation_strategy:
    - CONTINUE_INDEPENDENT_TASKS
    - CREATE_QUARANTINED_TEMPORARY_ARTIFACT
    - REASSESS_SUSPENDED_TASK
    - TERMINAL_FAILURE
```

规则：

```text
- 不依赖搁置任务的任务可以继续。
- 依赖搁置任务的任务默认暂停。
- 如果需要验证下游结构，可以创建临时隔离物。
- 临时隔离物必须由 Planner 定义，Dev 实现，Orchestrator 记录，Tester 最终验证。
```

### 12.2 临时隔离物定义

Planner 在 `doc/temporary-artifacts.md` 中定义：

```yaml
temporary_artifact_register:
  - temp_artifact_id: TEMP-001
    created_for_suspended_task: T-003
    purpose: 用于 T-004 的界面结构开发，临时代替未完成的数据层。
    allowed_usage:
      - 局部开发
      - 局部测试
    forbidden_usage:
      - 最终交付
      - 作为 confirmed-baseline 满足证据
    replacement_condition: T-003 恢复并完成后替换。
    must_remove_before_final: true
```

Dev 只能创建 Planner 明确允许的临时隔离物。

### 12.3 临时隔离物执行状态

Orchestrator 在 `execution-state.md` 中记录实际状态：

```yaml
temporary_artifact_execution_state:
  - temp_artifact_id: TEMP-001
    actual_files:
      - src/mocks/temp-user-api.ts
    created_by_agent_id: dev-a95e84
    associated_task_id: T-004
    current_status: active / replaced / removed / unresolved
```

### 12.4 临时隔离物最终检查

Final Completion Gate 中，Tester 必须检查：

```text
- 是否仍有 unresolved temporary artifacts。
- mock/stub/placeholder 是否被用于正式路径。
- temporary artifact 是否被用作冻结基线满足证据。
- must_remove_before_final 是否全部完成。
```

如果仍有 unresolved 临时物：

```text
Tester 输出 FAIL 或 BLOCKED。
Planner 不能 PROJECT_COMPLETE。
Planner 只能寻找替换路径，路径耗尽则 TERMINAL_FAILURE。
```

## 13. Final Completion Gate

### 13.1 触发

Final Completion Gate 只能由 Planner 发起。

触发条件：

```text
- Planner 判断没有更多可执行 next_action。
- 所有任务状态为 completed 或 suspended。
- 所有临时隔离物状态已知。
- 所有 suspended tasks 至少被重新评估过一次，或 Planner 明确说明其不影响最终交付。
```

Planner 输出：

```yaml
next_action:
  target_agent: tester-agent
  action: start_final_verification
  task_id: FINAL
  task_payload:
    task_title: Final Completion Gate
    required_context_files:
      - doc/confirmed-baseline.md
      - doc/plan.md
      - doc/execution-state.md
      - doc/temporary-artifacts.md
      - doc/dev-spec.md
```

### 13.2 Tester 最终验证

Tester 必须验证：

```text
- confirmed-baseline.md 是否全部满足。
- 所有 acceptance criteria 是否满足。
- 是否存在 unresolved temporary artifacts。
- 是否存在影响交付的 suspended tasks。
- 是否存在低质量通过或跳过验证。
- 整体功能、集成、视觉、交互、代码规范是否通过。
```

Tester 只输出事实，不宣告完成。

### 13.3 Planner 最终宣告

Planner 根据 Final Test Result Packet 输出最终决策。

`PROJECT_COMPLETE` 必须同时满足：

```text
- confirmed-baseline.md 全部满足。
- 所有 acceptance criteria 通过。
- 没有 unresolved temporary artifacts。
- 没有影响最终交付的 suspended tasks。
- 最终集成测试 PASS。
- 没有 LOW_QUALITY_PASS。
```

`TERMINAL_FAILURE` 满足任一条件即可：

```text
- 任一冻结基线条目无法满足。
- 任一必须替换的临时物 unresolved。
- 任一 suspended task 阻断最终交付。
- Tester 最终验证 FAIL 且 Planner 找不到不改基线的执行路径。
- 继续执行必然违反已确认需求、验收标准、技术约束、交付范围或已确认假设。
```

### 14.2 Dev Agent 模板

```yaml

name: dev-agent

你是 Dev Agent。你只负责实现 task_payload 授权的任务。

由 task_payload.required_context_files 指定。
```

```yaml
- PASS
```

## 15. 子智能体创建、Resume 与记录

### 15.1 创建原则

```text
- 每个任务首次开发：创建新的 Dev Agent。
- 同一任务修正：resume 原 Dev Agent。
- 每个任务首次测试：创建新的 Tester Agent。
- 同一任务复测：resume 原 Tester Agent。
- Planner 可以被多次调用，但它始终是计划权威。
```

### 15.2 Resume 原则

```text
- 普通 FAIL 修正时，必须 resume 同一个 Dev Agent。
- 同一任务复测时，应该 resume 同一个 Tester Agent。
- 三轮 FAIL 后，不继续 resume 原 Dev 进行第四轮修正。
- 三轮 FAIL 后，必须将完整事实证据包交给 Planner。
```

### 15.3 Agent ID 获取

在 Claude 子智能体环境中，可通过以下方式获取最近创建的子智能体 ID：

```bash
ls -lt ~/.claude/projects/*/*/subagents/agent-*.meta.json | head -3
```

从文件名提取裸 ID：

```text
agent-a95e84cd0b54c85ad.meta.json
→ a95e84cd0b54c85ad
```

备用：

```bash
find ~/.claude/projects -name "agent-*.meta.json" -type f -printf '%T@ %p\n' | sort -rn | head -5
```

### 15.4 子智能体输出读取

如 UI 未展示子智能体回复，可读取 JSONL：

```bash
cat ~/.claude/projects/{项目}/{会话}/subagents/agent-{ID}.jsonl | tail -50
```

这些操作属于运行接入细节，不改变协议权威边界。

```text
- 哪些路径被尝试过？
- 哪些假设已经冻结？
```

## 17. 防错清单

### 17.1 防止 Orchestrator 越权

```text
□ Orchestrator 是否自己写了任务拆分？如果是，错误。
□ Orchestrator 是否自己决定下一个任务？如果是，错误。
□ Orchestrator 是否自己总结失败根因？如果是，错误。
□ Orchestrator 是否自己补充 Dev/Tester prompt 的业务内容？如果是，错误。
□ Orchestrator 是否修改 doc/plan.md？如果是，错误。
□ Orchestrator 是否写业务代码或架构代码？如果是，错误。
```

### 17.2 防止 Planner 越过冻结基线

```text
□ Planner 是否 ASK_USER？如果是，错误。
□ Planner 是否修改冻结后的 confirmed-baseline.md？如果是，错误。
□ Planner 是否降低验收标准？如果是，错误。
□ Planner 是否新增或删除已确认交付范围？如果是，错误。
□ Planner 是否把临时物当成正式交付？如果是，错误。
□ Planner 是否输出 LOW_QUALITY_PASS？如果是，错误。
```

### 17.3 防止 Dev/Tester 越权

```text
□ Dev 是否修改计划文件？如果是，错误。
□ Dev 是否判断“这是计划问题”？如果是，错误。
□ Dev 是否建议重规划？如果是，错误。
□ Tester 是否修改代码？如果是，错误。
□ Tester 是否宣告项目完成？如果是，错误。
□ Tester 是否建议降低验收标准？如果是，错误。
```

### 17.4 防止临时物污染最终交付

```text
□ 是否存在 unresolved temporary artifacts？如果是，不能完成。
□ mock/stub/placeholder 是否仍在正式路径中？如果是，不能完成。
□ temporary artifact 是否被用作 confirmed-baseline 的满足证据？如果是，错误。
□ Tester 最终验收是否检查 temporary-artifacts.md？如果否，错误。
```

## 18. 快速落地清单

启动新项目时，按以下顺序执行。

```text
□ 1. 准备基础文件
   doc/requirements.md
   doc/dev-spec.md
   doc/lessons-learned.md

□ 2. Orchestrator 初始化
   mkdir -p doc logs logs/packets
   touch doc/plan.md
   touch doc/confirmed-baseline.md
   touch doc/execution-state.md
   touch doc/temporary-artifacts.md
   touch doc/user-clarifications.md
   touch doc/requirements-addendum.md
   touch logs/orchestrator-log.md

□ 3. 创建 Agent 模板
   .claude/agents/planner-agent.md
   .claude/agents/dev-agent.md
   .claude/agents/tester-agent.md

□ 4. 调用 Planner 进入 INITIAL_PLANNING
   Planner 读取需求、规范、经验库、代码结构
   Planner 不 ASK_USER
   Planner 写保守假设
   Planner 输出 Initial Approval Package

□ 5. 用户确认 Initial Approval Package
   确认完整计划
   确认 Assumption Register
   确认 Frozen Baseline Preview
   确认后所有假设进入 confirmed-baseline.md

□ 6. Planner 生成 confirmed-baseline.md
   冻结需求
   冻结验收标准
   冻结保守假设
   冻结技术约束和交付范围

□ 7. 进入 EXECUTION_LOOP
   Planner 输出 Plan Decision Packet
   Orchestrator 读取 next_action
   Dev 执行
   Tester 验证
   PASS → Planner 请求下一步
   FAIL < 3 → resume Dev
   FAIL = 3 → 交 Planner
   BLOCKED → 交 Planner

□ 8. 处理搁置和临时物
   Planner 输出 SUSPEND_TASK
   Planner 定义 temporary artifacts
   Orchestrator 记录执行状态
   Tester 最终验证是否清除

□ 9. Final Completion Gate
   Planner 发起最终验收
   Tester 输出 Final Test Result Packet
   Planner 宣告 PROJECT_COMPLETE 或 TERMINAL_FAILURE

□ 10. 归档
   完成报告或失败报告
   orchestrator-log.md
   packet 归档
   execution-state.md
```

## 19. 适用场景与边界

### 19.1 适用场景

```text
- 长时开发任务。
- 可拆分为多个可验证子任务的项目。
- 需要计划、开发、测试职责分离的项目。
- 需要强可追溯性的自动化流程。
- 不希望执行中频繁打断用户的项目。
- 能接受初始计划中保守假设被确认后冻结的项目。
```

### 19.2 不适用场景

```text
- 需求高度开放，需要持续和用户共创。
- 用户希望每次计划变化都确认。
- 无法接受自动做保守假设。
- 任务强耦合到无法局部验证。
- 失败后希望允许低质量通过。
- 项目目标本身尚不稳定。
```

## 20. 最终定义

本协议的本质是一套权力分配机制。

```text
Planner 拥有计划权，但不能写代码、不能问用户、不能改冻结基线。
Dev 拥有实现权，但不能改计划、不能解释失败根因。
Tester 拥有事实验证权，但不能修代码、不能宣告完成。
Orchestrator 拥有调度权，但不能思考计划、不能解释需求、不能判断根因。
User 拥有初始确认权，但不参与执行期微决策。
```

这套协议试图让每个角色都更窄、更清楚、更可审计。

一个运行良好的多智能体系统，不应该依赖某个智能体“刚好理解了全部上下文”。它应该依赖：

```text
- 明确的权威文件
- 明确的结构化消息
- 明确的状态机
- 明确的失败出口
- 明确的不可越权规则
```

因此，本协议的最终目标不是“让 AI 一次做完所有事”，而是让多智能体在长时任务中持续保持可控、可追溯、可终止。
