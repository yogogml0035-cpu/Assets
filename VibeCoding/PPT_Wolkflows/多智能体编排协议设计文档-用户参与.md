## 0. 文档目标

本文定义一套用于长时、多阶段、可验证任务的多智能体编排协议。它解决的问题不是“让一个智能体一次做完所有事”，而是建立一套可审计、可恢复、可重规划、可终止的协作系统。

协议的基本目标有五个：

1. **把计划权从主智能体中剥离出来**  
   主智能体不生成计划、不修改计划、不判断计划是否合理，只执行 Planner Agent 返回的结构化决策。

2. **把执行状态和计划内容彻底分离**  
   计划由 Planner 生成和维护；执行状态由主智能体机械记录。

3. **把用户参与限制在目标与边界确认上**  
   用户确认初始计划、冻结基线、需求补充和任何基线变更；用户不参与普通执行过程，也不参与最终结束判断。

4. **用结构化 Packet 替代自然语言流程控制**  
   所有 Planner、Dev、Tester、Integration Tester、Lessons Curator 的关键输出都必须落成结构化文件。

5. **禁止失败伪装成成功**  
   不允许“低质量通过”；如果无法满足冻结基线，系统只能声明未完成，或请求用户修改目标/边界。

## 1. 协议总览

### 1.1 系统角色

本协议包含七类角色：

```text
User
  │
  ▼
Orchestrator ── Planner Agent
  │                 │
  ├── Dev Agent      │
  ├── Tester Agent   │
  ├── Integration Tester
  └── Lessons Curator
```

角色之间不是平级协作，而是按照权限边界和协议状态机进行交互。

| 角色 | 中文名 | 核心定位 | 是否有计划权 | 是否可写代码 |
|---|---|---|---:|---:|
| User | 用户 | 确认目标与边界 | 否 | 否 |
| Orchestrator | 主智能体 / 编排器 | 机械调度、状态、落盘、schema 检查 | 否 | 否 |
| Planner Agent | 计划子智能体 | 唯一计划生成者与计划修正者 | 是 | 否 |
| Dev Agent | 开发智能体 | 执行开发任务 | 否 | 是 |
| Tester Agent | 测试智能体 | 单任务验证，只输出事实 | 否 | 否 |
| Integration Tester | 整体验收智能体 | 跨任务整体验收，只输出事实 | 否 | 否 |
| Lessons Curator | 经验沉淀智能体 | 维护正式经验库 | 否 | 否 |

### 1.2 权限关系图

```text
                    ┌────────────────────────────┐
                    │            User            │
                    │ 确认目标 / 边界 / 基线变更 │
                    └──────────────┬─────────────┘
                                   │
                                   ▼
┌──────────────────────────────────────────────────────────┐
│                      Orchestrator                         │
│ schema 检查 / 路由 / 状态 / 索引 / staging / promotion      │
│ 不解释需求，不判断计划，不写业务代码，不总结经验             │
└──────────────┬─────────────────────┬─────────────────────┘
               │                     │
               ▼                     ▼
     ┌──────────────────┐   ┌──────────────────────────────┐
     │  Planner Agent   │   │ Dev / Tester / Integration   │
     │ 唯一计划权威      │   │ 执行与事实证据输出            │
     └──────────────────┘   └──────────────────────────────┘
               │
               ▼
     ┌──────────────────┐
     │ Lessons Curator  │
     │ 正式经验沉淀      │
     └──────────────────┘
```

## 2. 系统级不可变原则

这些原则写入 `.claude/orchestrator-protocol.md`，所有项目都必须遵守。Planner 不能在项目策略中覆盖这些原则。

### 2.1 文件即记忆

所有影响后续判断的产物必须写入文件，包括：

- 计划版本；
- 执行状态；
- Planner 决策；
- Dev / Tester 结果包；
- 证据文件；
- 用户确认记录；
- 需求补充；
- 冻结基线；
- 经验候选与正式经验。

任何只存在于对话里的关键信息，都不具备协议效力。

### 2.2 隔离即常态

每个子智能体只能依赖协议明确提供的上下文、文件和 Packet。不得假设其他 Agent 的对话历史、隐含意图或未落盘信息。

### 2.3 记录即保险

每一次 Agent 调用都必须保存结构化结果包；每一次路由、拒收、重试、promote、replan、ASK_USER 都必须可追溯。

### 2.4 主智能体不是大脑

Orchestrator 不拥有以下权力：

```yaml
forbidden:
  - 生成计划
  - 修改计划内容
  - 判断计划是否合理
  - 判断失败根因
  - 判断是否需要重规划
  - 选择下一个任务
  - 写业务代码
  - 写架构代码
  - 总结经验为正式经验
  - 解释用户需求
  - 降低验收标准
  - 宣布项目完成
```

Orchestrator 只拥有以下机械能力：

```yaml
allowed:
  - 创建运行目录和空文件
  - 检查 Packet schema
  - 保存 Packet 和 evidence
  - 根据 status / decision_type / next_action 路由
  - 维护 execution-state.md
  - 维护日志和索引
  - 对 Planner 的 exact_file_writes 做白名单路径落盘
  - 将 staging 文件在用户确认后原样 promoted
```

### 2.5 Planner 是唯一计划权威

Planner Agent 拥有以下权力：

```yaml
authority:
  - 生成初始计划
  - 生成 next_action
  - 判断是否需要重规划
  - 拆分任务
  - 插入依赖任务
  - 重排执行路径
  - 维护计划版本和 changelog
  - 发起 ASK_USER
  - 生成 confirmed-baseline 草稿
  - 生成 orchestration-policy 草稿
  - 判断 COMPLETE_PROJECT / DECLARE_UNRESOLVED
```

Planner Agent 不拥有以下权力：

```yaml
forbidden:
  - 写业务代码
  - 修改测试代码
  - 直接执行开发实现
  - 降低已确认验收标准
  - 绕过 confirmed-baseline 宣布完成
  - 在未确认时修改冻结基线
```

### 2.6 禁止低质量通过

协议不允许以下状态：

```yaml
forbidden_terminal_states:
  - LOW_QUALITY_PASS
  - FORCE_PASS
  - ACCEPTED_WITH_KNOWN_BASELINE_VIOLATIONS
```

项目只能进入以下终态之一：

```yaml
terminal_states:
  - COMPLETE_PROJECT
  - DECLARE_UNRESOLVED
  - PROTOCOL_ERROR
```

## 3. 协议层与策略层

系统分为两层：系统级不可变协议与项目级可配置策略。

### 3.1 系统级不可变协议

路径：`.claude/orchestrator-protocol.md`

作用：

- 定义角色边界；
- 定义权限边界；
- 定义 Packet schema；
- 定义路由规则；
- 定义禁止事项；
- 定义主智能体能检查什么、不能检查什么；
- 定义 Planner、Dev、Tester、Integration Tester、Lessons Curator 的职责。

该文件不是每个项目自由生成的内容，Planner 不能改写它。

### 3.2 Agent 模板

路径：`.claude/agents/*.md`

每个 Agent 模板内嵌本角色专属协议摘录，而不是每次都塞入完整协议。

建议文件：

```text
.claude/agents/planner-agent.md
.claude/agents/dev-agent.md
.claude/agents/tester-agent.md
.claude/agents/integration-tester-agent.md
.claude/agents/lessons-curator-agent.md
```

### 3.3 项目级可配置策略

路径：`doc/orchestration-policy.md`

生成者：Planner  
确认者：User  
执行者：Orchestrator

该文件只配置当前项目的自动化策略，不能违反系统级协议。

示例：

```yaml
retry_policy:
  max_task_fail_rounds: 3
  max_integration_failures: 3
  max_project_replans_after_integration: 3
  max_lineage_total_fail_count: 5

completion_policy:
  allow_low_quality_pass: false
  require_integration_test_pass: true
  complete_project_decider: planner-agent

ask_user_policy:
  ask_user_only_for:
    - requirement_clarification
    - baseline_change
    - orchestration_policy_change
```

```yaml
execution_state_contains:

  - lineage_total_fail_count
  - current_dev_agent_id
  - current_tester_agent_id
  - latest_packet_paths
  - active_confirmation_request
  - protocol_retry_counts
```

## 5. 用户参与协议

用户只参与目标与边界确认，不参与执行和结束判断。

### 5.1 用户必须参与的事项

```yaml
user_must_confirm:
  - 初始 doc/plan.md
  - 初始 doc/confirmed-baseline.md
  - 初始 doc/orchestration-policy.md
  - Planner 发起 ASK_USER 的需求澄清
  - doc/requirements-addendum.md 的新增或修改
  - 任何会修改 confirmed-baseline.md 的事项
  - 任何会修改 orchestration-policy.md 的事项
```

### 5.2 用户不参与的事项

```yaml
user_does_not_confirm:
  - 普通任务推进
  - Dev/Test 修正循环
  - Planner 自动调整执行路径
  - 任务拆分、顺序重排、依赖插入
  - Integration Tester 整体测试
  - Planner 判断 COMPLETE_PROJECT
  - Planner 判断 DECLARE_UNRESOLVED
```

### 5.3 用户确认动作

用户面对 staging 草稿时，可以做三种动作：

```yaml
user_confirmation_actions:
  - APPROVE
  - REJECT_WITH_REASON
  - REQUEST_REVISION
```

Orchestrator 只原样记录用户反馈，不解释、不润色、不改写。

```yaml
confirmation_event:
  draft_path: doc/staging/confirmed-baseline-v001-draft.md
  user_action: REQUEST_REVISION
  user_raw_feedback: "这里不要把移动端适配作为交付范围。"
  recorded_by: orchestrator
  next_route: planner-agent
```

Planner 读取旧 draft 和用户原始反馈，生成新的 draft version。旧 draft 不删除，标记为 superseded。

## 6. Staging / Committed 双层落盘机制

### 6.1 需要 staging 的文件

以下文件必须先进入 `doc/staging/`，等待用户确认后再 promoted：

```yaml
stage_required:
  - 初始计划草稿
  - confirmed-baseline 草稿
  - orchestration-policy 草稿
  - requirements-addendum 草稿
  - baseline-affecting 变更草稿
  - policy-affecting 变更草稿
```

### 6.2 可直接 committed 的文件

以下文件可以由 Planner 通过 `exact_file_writes` 直接进入正式路径：

```yaml
commit_directly_allowed:
  - 执行路径调整后的新 plan version
  - plan-changelog.md
  - 非基线变更的任务拆分
  - 非基线变更的依赖插入
  - 非基线变更的任务顺序重排
```

### 6.3 Orchestrator 的落盘权限

Planner 不直接写文件。Planner 返回 `exact_file_writes`，Orchestrator 只按白名单机械落盘。

```yaml
orchestrator_file_write_rules:
  allowed_paths:
    - doc/staging/
    - doc/plans/
    - doc/plan.md
    - doc/plan-changelog.md
    - doc/confirmed-baseline.md
    - doc/orchestration-policy.md
    - doc/requirements-addendum.md
    - doc/lessons-learned.md
  forbidden_paths:
    - src/
    - tests/
    - package.json
    - any_code_or_config_not_explicitly_whitelisted
  forbidden_actions:
    - rewriting_planner_content
    - filling_missing_plan_fields
    - changing_file_content_before_write
    - deciding_semantic_correctness
```

### 6.4 Promotion 规则

用户批准 staging 草稿后，Orchestrator 原样 promote。

```yaml
promotion_event:
  source: doc/staging/confirmed-baseline-v001-draft.md
  target: doc/confirmed-baseline.md
  source_hash: sha256:...
  target_hash: sha256:...
  confirmed_by: user
  confirmation_log_ref: doc/confirmation-log.md#event-004
  promoted_by: orchestrator
```

## 7. 初始化协议

初始化阶段的目标是建立三个已确认的基础文件：

```text
1. 初始计划
2. 冻结基线
3. 项目编排策略
```

### 7.1 Orchestrator 初始化

Orchestrator 可以做运行环境级初始化：

```yaml
allowed_initialization:
  - 创建 doc/
  - 创建 logs/
  - 创建 runs/
  - 创建 doc/staging/
  - 创建 doc/plans/
  - 创建空的 execution-state.md
  - 创建空的 lesson-candidates.md
  - 创建空的 orchestrator-log.md
```

Orchestrator 不允许：

```yaml
forbidden_initialization:
  - 生成初始计划
  - 拆分任务
  - 搭建业务框架
  - 写公共组件
  - 选择技术路线
  - 写 Phase 0 代码
```

### 7.2 Planner 生成初始计划草稿

Planner 读取：

```text
- doc/requirements.md
- doc/dev-spec.md，如存在
- doc/lessons-learned.md，如存在
- 项目目录结构
- 已有代码，如项目不是空项目
```

Planner 输出：

```yaml
Plan Decision Packet:
  decision_type: STAGE_INITIAL_PLAN
  file_write_mode: STAGE_FOR_USER_CONFIRMATION
  exact_file_writes:
    - path: doc/staging/plan-v001-draft.md
      promotion_target: doc/plans/plan-v001.md
      content: ...
```

用户确认后，Orchestrator promote 到：

```text
doc/plans/plan-v001.md
doc/plan.md
```

### 7.3 Planner 生成冻结基线草稿

初始计划确认后，Planner 生成 `confirmed-baseline` 草稿。

冻结基线只冻结目标和边界，不冻结执行路径。

```yaml
confirmed_baseline_freezes:
  - 需求目标
  - 功能范围
  - 非功能要求
  - 核心验收标准
  - 明确技术约束
  - 禁止事项
  - 交付物边界

confirmed_baseline_does_not_freeze:
  - 任务顺序
  - 任务拆分方式
  - 中间实现步骤
  - 局部执行路径
  - 修正循环后的重排
```

### 7.4 Planner 生成项目编排策略草稿

Planner 生成 `doc/orchestration-policy.md` 草稿，用户确认一次。

策略可以配置：

```yaml
project_policy_can_configure:
  - max_task_fail_rounds
  - max_integration_failures
  - max_project_replans_after_integration
  - max_lineage_total_fail_count
  - final_phase_extra_checks
  - unresolved_report_requirements
```

策略不能配置：

```yaml
project_policy_cannot_override:
  - 主智能体不得生成计划
  - 主智能体不得写业务代码
  - Planner 不得写代码
  - Tester 不得改代码
  - 禁止 LOW_QUALITY_PASS
  - COMPLETE_PROJECT 必须基于 confirmed-baseline
```

### 7.5 初始化完成条件

只有以下条件同时满足，系统才能进入执行阶段：

```yaml
initialization_complete_when:
  - initial_plan_confirmed: true
  - confirmed_baseline_confirmed: true
  - orchestration_policy_confirmed: true
  - active_plan_version: plan-v001
  - project_status: READY_TO_EXECUTE
```

## 8. Planner 调用协议

### 8.1 Planner 默认无状态化

Planner 原则上每次调用都新建。

```yaml
planner_lifecycle:
  default: create_new_each_planning_event
  exception:
    - same_ASK_USER_clarification_round_can_resume
```

Planner 不依赖上一轮对话记忆，只依赖权威文件和结果包。

### 8.2 Planner 必读文件

每次 Planner 被调用时，必须读取或引用：

```text
- .claude/orchestrator-protocol.md 中 Planner 相关摘录
- doc/requirements.md
- doc/user-clarifications.md
- doc/requirements-addendum.md
- doc/confirmed-baseline.md
- doc/orchestration-policy.md
```

## 11. Orchestrator 执行协议

### 11.1 Orchestrator 只读字段

Orchestrator 只读取控制字段：

```yaml
read_for_routing:
  - packet_type
  - packet_version
  - agent_role
  - task_id
  - status
  - decision_type
  - next_action.target_agent
  - next_action.action
  - next_action.agent_lifecycle
  - next_action.resume_agent_id
  - file_write_mode
  - exact_file_writes.path
  - exact_file_writes.promotion_target
```

Orchestrator 不读取自由文本来判断语义。

### 11.2 Orchestrator 固定路由规则

```text
Planner decision_type = CONTINUE_CURRENT_PLAN / PATCH_PLAN / REGENERATE_PLAN
→ 按 next_action 调用目标 Agent

Dev status = DONE
→ 创建或 resume Tester，具体由 Planner next_action 或当前任务模板指定

Dev status = BLOCKED
→ 保存事实证据包
→ 交给 Planner

Tester status = PASS
→ 更新 execution-state
→ 交给 Planner 请求 next_action

Tester status = FAIL 且 current_task_fail_count < max_task_fail_rounds
→ resume Dev Agent

Tester status = FAIL 且 current_task_fail_count = max_task_fail_rounds
→ 保存三轮事实证据包
→ 交给 Planner

Tester status = BLOCKED
→ 保存事实证据包
→ 交给 Planner

Integration Tester status = PASS
→ 交给 Planner 判断 COMPLETE_PROJECT

Integration Tester status = FAIL / BLOCKED
→ 交给 Planner

Planner decision_type = COMPLETE_PROJECT
→ 标记 project_status = COMPLETED

Planner decision_type = DECLARE_UNRESOLVED
→ 标记 project_status = UNRESOLVED

Planner decision_type = ASK_USER
→ 原样展示问题并等待用户回答
```

### 11.3 不可变 Prompt 模板

Orchestrator 不写任务说明，只把 Planner 的 `task_payload` 套入不可变模板。

```text
你是 {target_agent}。

你必须执行下面 task_payload 中定义的任务。
你不得扩大或缩小任务范围。
你不得修改 task_payload 未授权的范围。
你必须按照本角色的 Agent Result Packet schema 输出。

<task_payload>
{Planner 提供的结构化任务包原文}
</task_payload>
```

Orchestrator 不补充验收标准，不解释任务目标，不增加上下文文件。

## 12. Dev Agent 协议

### 12.1 职责

Dev Agent 负责实现 Planner 分配的开发任务。

允许：

```yaml
allowed:
  - 读取 task_payload
  - 读取 required_context_files
  - 修改被授权的 target_files
  - 运行必要命令进行自测
  - 输出 AgentResultPacket
```

禁止：

```yaml
forbidden:
  - 判断是否需要重规划
  - 建议修改计划
  - 修改未授权文件
  - 修改 confirmed-baseline
  - 修改 orchestration-policy
  - 降低验收标准
```

### 12.2 状态枚举

```yaml
dev_status:
  - DONE
  - BLOCKED
```

Dev 不能输出 FAIL。开发完成但可能有问题时，仍然输出 DONE，并把不确定性写入 `uncertainties`。无法继续时输出 BLOCKED。

## 13. Tester Agent 协议

### 13.1 职责

Tester Agent 负责单任务验收。

允许：

```yaml
allowed:
  - 读取任务上下文
  - 读取代码
  - 运行测试命令
  - 截图或读取测试结果
  - 输出事实证据包
```

禁止：

```yaml
forbidden:
  - 修改代码
  - 给出重规划建议
  - 判断计划根因
  - 新增任务建议
  - 调整验收标准
```

### 13.2 状态枚举

```yaml
tester_status:
  - PASS
  - FAIL
  - BLOCKED
```

`FAIL` 表示任务未满足当前验收标准。  
`BLOCKED` 表示无法完成验收，例如缺少运行条件、缺少必要文件、验收标准不可验证。

## 14. 修正循环协议

普通任务失败先视为实现问题，进入最多 3 轮 Dev/Test 修正循环。

### 14.1 普通 FAIL

```text
Tester FAIL
  │
  ├─ current_task_fail_count < max_task_fail_rounds
  │     → resume same Dev Agent
  │     → Dev 修正
  │     → resume same Tester Agent
  │
  └─ current_task_fail_count = max_task_fail_rounds
        → 保存三轮事实证据包
        → 交给 Planner 判断
```

Orchestrator 不判断失败原因，只计数和路由。

### 14.2 BLOCKED

```text
Dev / Tester status = BLOCKED
  → 不走 3 轮修正
  → 保存事实证据包
  → 交给 Planner
```

### 14.3 Planner 介入后

Planner 可返回：

```yaml
planner_after_failure:
  - CONTINUE_CURRENT_PLAN
  - PATCH_PLAN
  - REGENERATE_PLAN
  - ASK_USER
  - DECLARE_UNRESOLVED
```

## 15. Agent 生命周期协议

### 15.1 Dev / Tester 生命周期

```yaml
default_agent_lifecycle:
  new_task_first_execution: create_new
  same_task_fail_fix: resume_existing
  same_task_retest: resume_existing
  new_task_after_split: create_new
  new_task_after_replan: create_new
  dependency_inserted_task: create_new
```

但最终由 Planner 在 `next_action.agent_lifecycle` 中显式声明。

```yaml
next_action:
  target_agent: dev-agent
  action: resume_current_task
  task_id: T-04
  agent_lifecycle: resume_existing
  resume_agent_id: abc123
```

Orchestrator 不判断“是否同一任务”，只执行字段。

### 15.2 Planner 生命周期

Planner 默认每次规划事件新建；同一 `ASK_USER` 澄清回合内可 resume。

### 15.3 Integration Tester 生命周期

Integration Tester 通常每次整体验收新建，确保它只依赖文件和证据，不依赖上一轮对话记忆。

### 15.4 Lessons Curator 生命周期

Lessons Curator 可按事件或阶段新建；正式经验必须依据 evidence_refs。

### 18.1 基线内容

```yaml
confirmed_baseline:
  confirmed_requirements:

      statement: "..."

  confirmed_constraints:
```

## 19. 需求澄清协议

当 Planner 发现需求不清、冲突或缺少关键决策时，返回 `ASK_USER`。

### 19.1 原始澄清记录

Orchestrator 维护：

```text
doc/user-clarifications.md
```

只记录 Planner 问题和用户原始回答，不总结、不润色、不解释。

格式：

```markdown
## Q-2026-04-30-001

### 用户原始回答
```

Planner 维护：

```text
doc/requirements-addendum.md
```

## 20. 整体验收与项目结束协议

### 20.1 整体测试必须是显式阶段

Planner 必须在初始计划或后续计划版本中显式安排整体测试阶段。

```yaml
final_phase:
  - task_id: T-FINAL-01
    type: integration_test
    owner: integration-tester-agent
    goal: "验证整个项目是否满足 confirmed-baseline.md"
    required_context:
      - doc/confirmed-baseline.md
      - doc/requirements.md
      - doc/requirements-addendum.md
      - doc/plans/{active_plan_version}.md
      - doc/execution-state.md
      - runs/**/result.yaml
      - doc/lessons-learned.md
```

### 20.2 Integration Test Result Packet

```yaml
packet_type: IntegrationTestResultPacket
packet_version: "1.0"
agent_role: integration-tester-agent
status: PASS      # PASS / FAIL / BLOCKED

task_id: T-FINAL-01
active_plan_version: plan-v006
confirmed_baseline_ref: doc/confirmed-baseline.md

baseline_coverage:
  - baseline_item_id: AC-001
    result: satisfied      # satisfied / violated / unverified
    evidence_refs:
      - runs/T-FINAL-01/evidence/001-output.txt

cross_task_issues:
  - issue_id: X-001
    observed_fact: "..."
    affected_task_ids:
      - T-02
      - T-05
    evidence_refs:
      - runs/T-FINAL-01/evidence/002-screenshot.png

uncertainties:
  - "..."
```

### 20.3 整体测试结果路由

```text
Integration Tester PASS
→ Orchestrator 保存结果包
→ 调用 Planner
→ Planner 判断 COMPLETE_PROJECT 或 ASK_USER

Integration Tester FAIL / BLOCKED
→ Orchestrator 保存结果包
→ 调用 Planner
→ Planner 返回 PATCH_PLAN / REGENERATE_PLAN / ASK_USER / DECLARE_UNRESOLVED
```

Integration Tester 不判断下一步，不指定 Dev 修什么，不宣布项目结束。

### 20.4 项目完成条件

Planner 只有在满足以下条件时才能返回 `COMPLETE_PROJECT`：

```yaml
complete_project_allowed_when:
  - integration_test_status: PASS
  - confirmed_baseline_satisfied: true
  - unresolved_critical_issues: 0
  - no_baseline_violations: true
```

用户不做最终确认。

### 20.5 未完成声明

超过项目级终止阈值后，Planner 只能返回：

```yaml
allowed_after_limits_reached:
  - ASK_USER
  - DECLARE_UNRESOLVED
```

`DECLARE_UNRESOLVED` 必须生成未完成报告：

```yaml
unresolved_report:
  unsatisfied_baseline_items:
    - AC-003
  attempted_plan_versions:
    - plan-v004
    - plan-v005
    - plan-v006
  failed_lineages:
    - L-T04-001
  key_evidence_refs:
    - runs/T-FINAL-01/evidence/002-error-log.txt
  planner_explanation: "..."
```

## 21. 项目级终止阈值

项目级终止阈值写入 `doc/orchestration-policy.md`。

示例：

```yaml
project_level_limits:
  max_task_fail_rounds: 3
  max_integration_failures: 3
  max_project_replans_after_integration: 3
  max_lineage_total_fail_count: 5
```

超过阈值不代表成功，也不代表自动失败。超过阈值后，Planner 必须在以下两类动作中选择：

```yaml
planner_choices_after_limit:
  - ASK_USER
  - DECLARE_UNRESOLVED
```

如果 `ASK_USER`，含义是当前目标/边界可能不可达，需要用户重新确认或修改基线。  
如果 `DECLARE_UNRESOLVED`，含义是系统明确承认未完成，并生成未完成报告。

## 22. Lessons Curator 协议

### 22.1 经验候选池

Dev / Tester / Planner / Integration Tester 都可以在结果包中输出 `lessons_candidates`。

Orchestrator 只机械追加到：

```text
doc/lesson-candidates.md
```

Orchestrator 不总结、不改写、不判断价值。

### 22.2 正式经验库

Lessons Curator 维护：

```text
doc/lessons-learned.md
```

正式经验必须满足：

```yaml
lesson_entry_required_fields:
  - lesson_id
  - category
  - reusable_rule
  - source_task_id
  - evidence_refs
  - created_at
```

### 22.3 Lessons Curator 调用时机

采用事件驱动 + 阶段沉淀。

立即触发：

```yaml
trigger_immediately:
  - BLOCKED
  - 三轮 FAIL 后交给 Planner
  - Planner 发生重规划
  - 用户澄清导致需求补充
  - 整体测试发现跨任务问题
```

阶段性触发：

```yaml
trigger_by_phase:
  - 每个 Phase 完成后
  - 全部任务完成后
  - 项目结束前最终整理一次
```

Lessons Curator 禁止修改计划、基线、策略和代码。

## 23. Packet 合规检查协议

### 23.1 Orchestrator 可检查范围

```yaml
schema_validation_scope:
  - required_fields_present
  - enum_values_valid
  - packet_type_matches_agent_role
  - target_agent_valid
  - task_id_present_when_required
  - forbidden_fields_absent
  - file_paths_within_whitelist
  - required_file_refs_present
```

### 23.2 Orchestrator 禁止检查范围

```yaml
semantic_validation_forbidden:
  - 判断计划是否合理
  - 判断失败根因
  - 判断 Planner 是否应该重规划
  - 判断需求解释是否正确
  - 阅读自由文本识别暗示性越权
  - 从自然语言中推断缺失字段
```

### 23.3 不合规 Packet 处理

```text
第 1 次不合规
→ 保存 invalid packet
→ resume 同一个 Agent
→ 要求“只修正格式，不改变事实内容”

第 2 次仍不合规
→ 再次保存 invalid packet
→ 最后一次格式重试

第 3 次仍不合规
→ 进入 PROTOCOL_ERROR 或交给 Planner，取决于 Agent 类型
```

更细规则：

```yaml
invalid_packet_policy:
  planner_invalid:
    retry_same_agent: 2
    after_retry: project_status_PROTOCOL_ERROR
  dev_or_tester_invalid:
    retry_same_agent: 2
    after_retry: send_invalid_packets_to_planner
  lessons_curator_invalid:
    retry_same_agent: 2
    after_retry: skip_current_curation_and_log
```

### 23.4 语义越权处理

采用三层机制：

```yaml
semantic_overreach_handling:
  1_fact_field_constraints:
    - 只允许事实、证据、尝试步骤、影响、不确定性
  2_forbidden_fields:
    - 禁止 replan_suggestion 等字段名
  3_planner_filtering:
    - Planner 只采纳事实证据，不采纳非 Planner 的建议
```

默认不启用 Protocol Guardian；高风险项目可额外启用。

## 25. 权限设计

### 25.1 工具权限建议

| Agent | Read | Glob/Grep | Bash | Edit/Write | 说明 |
|---|---:|---:|---:|---:|---|
| Orchestrator | 是 | 是 | 可选 | 仅白名单落盘 | 不写业务代码 |
| Planner | 是 | 是 | 可选 | 否 | 通过 exact_file_writes 让 Orchestrator 落盘 |
| Dev | 是 | 是 | 是 | 是 | 只改 task_payload 授权范围 |
| Tester | 是 | 是 | 是 | 否 | 只读测试 |
| Integration Tester | 是 | 是 | 是 | 否 | 只读整体验收 |
| Lessons Curator | 是 | 是 | 否/可选 | 仅 lessons 文件 | 不改计划和代码 |

### 25.2 Planner 无直接写权限的理由

如果工具权限是 `Edit / Write` 级别，而不是路径级别，那么 Planner 拿到写权限后理论上可以改代码。为保证边界，Planner 只生成 `exact_file_writes`，由 Orchestrator 按路径白名单原样落盘。

关键定义：

> Orchestrator 写入 Planner 生成的文件内容，不等于 Orchestrator 参与生成计划。它只是受限落盘执行器。

## 26. Agent 模板要求

### 26.1 Planner Agent 模板必须包含

```yaml
must_include:
  - 你是唯一计划权威
  - 你不得写代码
  - 你必须读取 confirmed-baseline
  - 你必须输出 PlanDecisionPacket
  - 你必须声明 baseline_impact
  - 你必须给出 next_action
  - 你必须通过 exact_file_writes 请求文件落盘
```

### 26.2 Dev Agent 模板必须包含

```yaml
must_include:
  - 你只执行 task_payload
  - 你不得判断是否重规划
  - 你不得建议新增任务
  - 你不得降低验收标准
  - 你必须输出 AgentResultPacket
  - status 只能是 DONE / BLOCKED
```

### 26.3 Tester Agent 模板必须包含

```yaml
must_include:
  - 你是只读角色
  - 你不得修改代码
  - 你不得判断计划问题
  - 你不得建议重规划
  - 你只输出事实、证据、问题和不确定性
  - status 只能是 PASS / FAIL / BLOCKED
```

### 26.4 Integration Tester 模板必须包含

```yaml
must_include:
  - 你只做整体验收
  - 你必须以 confirmed-baseline 为准
  - 你不得宣布项目完成
  - 你不得指定修复方案
  - 你必须输出 IntegrationTestResultPacket
```

### 26.5 Lessons Curator 模板必须包含

```yaml
must_include:
  - 你只维护正式经验库
  - 你不得修改计划、基线、策略、代码
  - 每条经验必须引用 evidence_refs
  - 不沉淀一次性细节
```

## 27. 典型流程剧本

### 27.1 正常任务通过

```text
Planner → next_action: start_new_task T-01 dev-agent
Orchestrator → 创建 Dev
Dev → DONE
Orchestrator → 创建 Tester
Tester → PASS
Orchestrator → 保存结果，更新 execution-state
Orchestrator → 调用 Planner
Planner → next_action: start_new_task T-02
```

### 27.2 普通失败三轮后重规划

```text
Tester → FAIL round 1
Orchestrator → resume Dev
Tester → FAIL round 2
Orchestrator → resume Dev
Tester → FAIL round 3
Orchestrator → 保存三轮结果包
Orchestrator → 调用 Planner
Planner → PATCH_PLAN: split T-04 into T-04A/T-04B
Orchestrator → 落盘 plan-v003
Orchestrator → 按 next_action 创建 Dev for T-04A
```

### 27.3 BLOCKED

```text
Dev → BLOCKED with evidence packet
Orchestrator → 不修 3 轮
Orchestrator → 调用 Planner
Planner → insert_dependency T-03.5
Orchestrator → 创建 Dev for T-03.5
```

### 27.4 需求不清

```text
Planner → ASK_USER Q1
Orchestrator → 原样展示问题
User → 回答
Orchestrator → 写 doc/user-clarifications.md
Planner → 生成 requirements-addendum draft
Orchestrator → staging 并展示
User → APPROVE
Orchestrator → promote
Planner → 更新计划或 next_action
```

### 27.5 整体测试失败

```text
Planner → RUN_INTEGRATION_TEST
Integration Tester → FAIL with baseline coverage evidence
Orchestrator → 调用 Planner
Planner → PATCH_PLAN / REGENERATE_PLAN / ASK_USER / DECLARE_UNRESOLVED
```

### 27.6 项目完成

```text
Integration Tester → PASS
Orchestrator → 调用 Planner
Planner → COMPLETE_PROJECT
Orchestrator → project_status = COMPLETED
```

用户不需要最终确认。

## 31. 术语表

| 术语 | 含义 |
|---|---|
| Orchestrator | 主智能体；协议执行器，不是计划者 |
| Planner Agent | 唯一计划生成、解释、修正者 |
| Plan Decision Packet | Planner 控制流程的唯一结构化输出 |
| Agent Result Packet | Dev / Tester 等 Agent 的结构化输出 |
| confirmed-baseline | 用户确认的目标与边界冻结文件 |
| orchestration-policy | 当前项目的调度、重试、终止策略 |
| staging | 等待用户确认的草稿区 |
| committed | 已生效的正式文件区 |
| promotion | 用户确认后从 staging 提升到正式路径 |
| lineage | 任务拆分或重规划后的问题血缘链 |
| BLOCKED | 无法继续执行或无法验证，不进入普通修正循环 |
| DECLARE_UNRESOLVED | 无法满足冻结基线时的明确未完成声明 |
| LOW_QUALITY_PASS | 禁止状态；不能把未满足基线的结果包装为完成 |

## 32. 协议一句话总结

本协议的核心不是“让主智能体更聪明”，而是**让主智能体更不聪明**：它只做可验证的机械执行，把计划、判断、开发、测试、经验沉淀拆给各自的权威角色，并用文件、Packet、版本、基线和状态机保证整个长时工作过程可追溯、可恢复、可终止。
