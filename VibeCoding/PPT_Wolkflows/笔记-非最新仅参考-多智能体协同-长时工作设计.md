## 一、核心原则
### 1.1 Harness Engineering 的本质
不是让 AI 一次做完所有事，而是设计一套"套件(harness)"——决定存储什么、检索什么、展示给模型什么。主智能体就是这个套件的编排者。
**核心矛盾**：上下文窗口有限 vs 任务复杂度无限。解法不是更大的窗口，而是更聪明的上下文管理。

### 1.2 三条铁律

| 铁律 | 含义 | 违反后果 |
|------|------|----------|
| **文件即记忆** | 子智能体的所有产出必须持久化到文件，不依赖内存 | 上下文丢失后无法恢复 |
| **隔离即常态** | 每个子智能体只看到主智能体给它的信息，不假设它知道外部状态 | 隐式依赖导致幻觉 |
| **记录即保险** | 主智能体和子智能体都写日志，确保一切可追溯 | 出问题后无法复盘 |

## 二、角色体系设计

### 2.1 三角色模型

```
┌─────────────┐
│   主智能体    │  编排者：拆任务、管状态、中转信息
│  (Orchestrator) │
└──────┬──────┘
       │
   ┌───┴───────────────┐
   │                   │
   ▼                   ▼
┌──────────┐     ┌──────────┐
│ 开发智能体 │     │ 测试智能体 │
│  (Dev)   │     │ (Tester)  │
└──────────┘     └──────────┘
```

**各角色核心职责**：

| 角色 | 职责 | 工具权限 | 上下文管理 |
|------|------|----------|------------|
| **主智能体** | 拆任务、管计划、中转信息、写日志 | 全部 | 保持全程上下文 |
| **开发智能体** | 读需求→写代码→自测→输出 | 读写 + Bash | 每个任务新创建或 resume |
| **测试智能体** | 读需求→审查代码/截图→输出结论 | 只读 + 截图工具 | 每个任务新创建或 resume |

### 2.2 主智能体：编排者

主智能体是整个流程的大脑，它需要做到：

1. **管理开发计划**：维护一个计划文件，记录每个子任务的状态（待办/进行中/已完成）
2. **管理经验库**：维护一个踩坑记录文件，每个子智能体完成后更新
3. **中转信息**：接收测试智能体的反馈，传递给开发智能体
4. **记录日志**：全程记录每个关键事件（时间、子智能体ID、结果）
5. **决策逻辑**：测试通过→下一任务；测试失败→循环修正

### 2.3 开发智能体：执行者

**核心原则**：每个子任务创建一个新子智能体，或 resume 上一个同任务的开发智能体。

为什么不用一个长对话的开发智能体？
- 上下文窗口会被历史占满
- 长对话后模型注意力分散
- 出错后难以定位问题

为什么 resume 而不是新建？
- 测试智能体的反馈需要结合开发时的上下文才能理解
- resume 让开发智能体能"看到"自己之前写了什么

**结论**：每个子任务的第一次开发用新智能体，后续修正用 resume 回同一智能体。

### 2.4 测试智能体：质检员

**核心原则**：只读，不修改任何代码文件。这是防止测试智能体"越权"的关键约束。

**测试维度的设计原则**：
- **可量化**：每项检查有明确的通过/失败标准（如"字号 ≥ 24px"）
- **可截图**：视觉测试必须基于截图，不依赖文字描述
- **分优先级**：严重/中等/轻微，明确什么算通过、什么算失败

## 三、工作流程

### 3.1 整体流程图

```
开始
  │
  ▼
[主智能体] 制定开发计划 → 输出：plan.md
  │
  ▼
[主智能体] 搭建初始框架和公共代码
  │
  ▼
═══════════ 逐任务循环 ═══════════
  │
  ├─► [开发智能体] 新建，执行任务 N
  │     ├─ 读取：需求、规范、经验库、已有代码
  │     ├─ 开发 + 自测
  │     └─ 输出：代码文件 + 完成摘要
  │
  ├─► [测试智能体] 新建，审查任务 N
  │     ├─ 读取：需求、规范、待测代码
  │     ├─ 截图 + 视觉分析 + 代码审查
  │     └─ 输出：PASS / FAIL + 问题列表
  │
  ├─► 测试结果？
  │     ├─ PASS → 更新计划，进入任务 N+1
  │     └─ FAIL → 进入修正循环 ↓
  │
  ├─► [开发智能体] resume，根据反馈修正
  │     └─ 输出：修改后的代码 + 更新摘要
  │
  ├─► [测试智能体] resume，重新审查
  │     └─ 输出：PASS / FAIL
  │
  └─► 回到"测试结果？"判断（最多循环 5 轮）
═══════════════════════════════════
  │
  ▼
全部任务完成 → 整体测试 → 结束
```

### 3.2 逐任务执行模板

主智能体对每个子任务的标准操作序列：

```
第 N 个任务：

1. 更新计划文件：任务 N 状态 → 进行中
2. 创建开发子智能体：
   Agent(
     subagent_type: "dev-agent",
     prompt: "
       ## 任务：{任务标题}
       ## 需求文件：doc/xxx.md 的第 N 节
       ## 必读文件：
       - doc/plan.md（开发计划，了解全局位置）
       - doc/visual-guide.md（设计需求）
       - doc/lessons-learned.md（经验教训）
       - src/（已有代码结构）
       ## 输出：代码文件路径 + 完成摘要
     "
   )
3. 等待开发完成，记录到日志
4. 获取子智能体 ID（见 4.1 节）
5. 创建测试子智能体：
   Agent(
     subagent_type: "tester-agent",
     prompt: "
       ## 审查任务：{任务标题}
       ## 待测文件：{开发智能体输出的路径}
       ## 需求文件：doc/xxx.md 的第 N 节
       ## 输出：PASS/FAIL + 问题列表
     "
   )
6. 等待测试完成，记录到日志
7. 获取子智能体 ID
8. 判定结果：
   - PASS → 更新计划：任务 N → 已完成
   - FAIL → 进入修正循环（见 3.3 节）
```

### 3.3 修正循环

```
修正循环第 K 轮（K ≤ 5）：

1. resume 开发智能体：
   Agent(
     resume: "{开发智能体裸ID}",
     subagent_type: "dev-agent",
     prompt: "
       测试反馈如下：
       {测试智能体的完整反馈}

       请根据反馈修改代码。输出修改后的文件路径和修改摘要。
     "
   )

2. resume 测试智能体：
   Agent(
     resume: "{测试智能体裸ID}",
     subagent_type: "tester-agent",
     prompt: "
       开发智能体已修改代码，请重新审查。
       待测文件：{路径}
       输出：PASS/FAIL + 问题列表
     "
   )

3. 判定：
   - PASS → 退出循环
   - FAIL 且 K < 5 → 继续循环
   - FAIL 且 K = 5 → 强制通过，标记为"低质量通过"
```

## 四、关键操作细节

### 4.1 子智能体 ID 获取

**每次创建子智能体后，立即获取其真实 ID**：

```bash
ls -lt ~/.claude/projects/*/*/subagents/agent-*.meta.json | head -3
```

从文件名提取：`agent-a95e84cd0b54c85ad.meta.json` → 裸ID = `a95e84cd0b54c85ad`

**备用方案**（如果上面不行）：
```bash
find ~/.claude/projects -name "agent-*.meta.json" -type f -printf '%T@ %p\n' | sort -rn | head -5
```

### 4.2 子智能体 Resume

```javascript
// 恢复开发智能体
Agent(
  resume: "a95e84cd0b54c85ad",
  subagent_type: "dev-agent",
  prompt: "新指令（如：根据测试反馈修改代码）"
)

// 恢复测试智能体
Agent(
  resume: "b73f21e0c98d12a4e",
  subagent_type: "tester-agent",
  prompt: "新指令（如：重新审查修改后的代码）"
)
```

**关键要点**：
- `resume` 参数填裸 ID，不是完整文件名
- `subagent_type` 必须与创建时一致
- resume 后子智能体能看到完整的历史对话

### 4.3 查看子智能体回复

子智能体的回复可能不在 UI 显示，需要读 JSONL 文件：

```bash
# 查看最近的子智能体输出
cat ~/.claude/projects/{项目}/{会话}/subagents/agent-{ID}.jsonl | tail -50
```

## 五、上下文管理策略

### 5.1 核心矛盾

| 方案 | 优点 | 缺点 |
|------|------|------|
| **一个长对话** | 上下文完整 | 窗口爆满，注意力分散 |
| **每任务新智能体** | 上下文干净 | 丢失历史，重复阅读 |
| **新智能体 + 文件传递** | 干净且有记录 | 需要精心设计传入内容 |

**推荐方案**：新智能体 + 精心设计的文件传递 + 经验库积累。

### 5.2 每个子智能体必须读取的上下文

开发智能体的必读清单：

```
1. 任务描述（主智能体传入的 prompt）     → 知道做什么
2. 需求文档（对应的 page/section）       → 知道做成什么样
3. 开发规范（skill 或规范文件）          → 知道怎么写
4. 已完成的代码结构（src/ 目录）         → 知道已有的模式
5. 经验教训库（lessons-learned.md）      → 知道别人踩过什么坑
6. 开发计划（plan.md）                  → 知道自己在全局中的位置
```

测试智能体的必读清单：

```
1. 任务描述（主智能体传入的 prompt）     → 知道测什么
2. 需求文档（对应的 page/section）       → 知道验收标准
3. 待测代码文件                         → 审查对象
4. 开发规范                             → 知道合规标准
5. 主题/样式配置（如 theme.ts）          → 视觉基准
```

### 5.3 经验库设计

经验库是跨子智能体传递知识的唯一可靠通道。

**文件路径**：`doc/lessons-learned.md`

**格式**：

```markdown
# 经验教训库

## 类别一：布局
- [04-05] 标题和正文字号差距至少 8px，否则层次不清
```

**经验总结的粒度原则**：
- ❌ 太细："Page03 第 42 行的 left 值应该是 120 而不是 100" → 只适用于一个像素
- ❌ 太宽："注意布局要好看" → 没有可操作性
- ✅ 恰好："标题和正文字号差距至少 8px" → 有具体标准，适用于所有场景

**判断标准**：如果一条经验能帮到下一个子智能体避免类似错误，就是好的粒度。

### 5.4 开发计划文件

**文件路径**：`doc/plan.md`

**格式**：

```markdown
# 开发计划

## 项目概述
{一句话描述项目目标}

## 任务列表
| # | 任务 | 状态 | 开发ID | 测试ID | 备注 |
|---|------|------|--------|--------|------|
| 0 | 搭建框架和公共代码 | ✅ 已完成 | - | - | 初始搭建 |
| 1 | Page01 - 标题页 | ✅ 已完成 | abc123 | def456 | 1轮通过 |
| 2 | Page02 - 内容页 | 🔄 进行中 | ghi789 | - | 第1次开发 |
| 3 | Page03 - 图表页 | ⏳ 待办 | - | - | - |
| 4 | 整体测试 | ⏳ 待办 | - | - | - |

## 当前进度
- 正在执行：任务 2（Page02）
- 已完成：2/4
```

## 六、子智能体模板设计

### 6.1 模板通用结构

一个子智能体模板文件（`.claude/agents/xxx.md`）应包含以下部分：

```yaml
---
name: agent-name           # 智能体名称
description: |             # 触发场景描述
  触发条件...
tools: Read, Edit, Write, Bash, Glob, Grep  # 工具权限
model: inherit / haiku / sonnet / opus       # 模型选择
permissionMode: acceptEdits  # 权限模式
memory: project              # 记忆模式
skills:                      # 预加载 skill
  - skill-name
---

## 角色定义
{一句话说明你是谁、干什么}

## 工作流程
{按步骤列出执行过程}

## 必读文件（按顺序）
{列出每次任务开始时必须读取的文件}

## 输出格式
{强制要求的输出模板}

## 经验积累
{完成后的经验记录方式}
```

### 6.2 模型选择原则

| 场景 | 推荐模型 | 理由 |
|------|----------|------|
| 简单、重复性任务 | haiku | 快速、便宜、够用 |
| 代码开发 | inherit（跟随主智能体） | 开发需要最强推理 |
| 测试/审查 | inherit 或 sonnet | 需要足够的判断力 |
| 关键决策 | opus | 最强推理能力 |

### 6.3 权限设计

| 角色 | 工具权限 | 理由 |
|------|----------|------|
| **开发智能体** | Read + Edit + Write + Bash + Glob + Grep | 需要读写代码、运行命令 |
| **测试智能体** | Read + Bash + Glob + Grep | 只读代码，Bash 用于截图/运行测试 |
| **测试智能体禁止** | Edit, Write | 防止测试智能体直接修改代码 |

### 6.4 开发智能体模板（通用版）

```yaml
---
name: dev-agent
description: |
  开发智能体。接收任务描述，读取上下文，开发代码，输出结果。
  触发场景：
  - "开发任务 N"
  - "修改/优化指定模块"
  - 需要编写或修改代码时
tools: Read, Edit, Write, Bash, Glob, Grep
model: inherit
permissionMode: acceptEdits
memory: project
---

你是开发工程师，负责根据规范开发高质量代码。

## 工作流程

### 1. 读取任务信息
- 确认任务编号和标题
- 确认是否有特殊要求

### 2. 读取必读文件（按顺序）
1. 开发计划文件 → 了解全局位置和当前待办
2. 需求文档（对应 section）→ 知道做成什么样
3. 开发规范文件 → 知道怎么写
4. 经验教训库 → 知道踩过的坑
5. 已有代码结构 → 知道已有的模式

### 3. 开发实现
- 遵循开发规范
- 复用已有组件和模式
- 自测确认无编译错误

### 4. 输出格式（必须严格遵守）

## 开发完成

### 文件路径
- {输出的代码文件}

### 实现摘要
- {列出实现的主要内容}

### 需要注意的问题（如有）
- {遗留问题或设计选择说明}
```

### 6.5 测试智能体模板（通用版）

```yaml
---
name: tester-agent
description: |
  测试智能体。审查代码质量和视觉效果，输出结构化测试报告。
  触发场景：
  - "测试任务 N"
  - "审查指定模块"
  - 开发完成后需要质量验证
tools: Read, Bash, Glob, Grep
disallowedTools: Edit, Write
model: inherit
memory: project
---

你是质量测试工程师，负责审查代码和视觉呈现。你是只读角色，**不修改任何代码文件**。

## 工作流程

### 1. 读取任务信息
### 2. 读取参考文件（需求 + 待测代码 + 规范）
### 3. 执行测试
### 4. 输出结构化报告

## 检查清单

| 维度 | 检查项 | 通过标准 |
|------|--------|----------|
| 需求匹配 | 核心功能是否实现 | 全部实现 |
| 代码规范 | 是否符合开发规范 | 无违规 |
| 视觉质量 | 截图分析 | 符合设计意图 |

## 判定标准
- PASS：所有维度通过，最多 1-2 个轻微问题
- FAIL：存在严重问题，或中等问题 ≥ 2 个

## 输出格式

### 检查结果
| 维度 | 结果 | 备注 |
|------|------|------|
| ... | ✅/❌ | ... |

### 问题列表（如有）

### 总体评价
一句话总结
```

## 七、开发计划拆分策略

### 7.1 拆分粒度

| 太粗 | 恰好 | 太细 |
|------|------|------|
| "做完整套动画" | "开发 Page03 - 数据图表页" | "写 Page03 第 12 行的 interpolate" |
| 一个任务 > 500 行代码 | 一个任务 100-300 行代码 | 一个任务 < 30 行代码 |
| 难以测试 | 可独立验证 | 上下文切换成本太高 |

**黄金法则**：一个子任务 = 一个可独立运行和验证的功能单元。

### 7.2 拆分顺序

```
Phase 0: 框架搭建
  → 项目初始化、公共组件、主题配置、路由配置
  → 由主智能体直接完成（不需要子智能体）

Phase 1~N: 逐功能开发
  → 每个功能一个子任务
  → 每个子任务走完整的开发-测试循环

Phase Last: 整体集成测试
  → 所有功能完成后，做一次整体测试
  → 关注跨功能的衔接和全局一致性
```

### 7.3 依赖关系处理

- 如果任务 B 依赖任务 A 的产出，A 必须先完成并通过测试
- 公共组件（如主题、工具函数）应该在 Phase 0 就完成
- 如果发现新依赖，暂停当前任务，先完成依赖项

## 八、日志与记录体系

### 8.1 三层日志架构

```
主智能体日志（全流程）
  │
  ├── 子智能体日志（各自独立）
  │     ├── 开发智能体日志
  │     └── 测试智能体日志
  │
  └── 经验库（跨任务积累）
```

### 8.2 主智能体日志格式

```markdown
# {项目名} 开发日志

## 初始化
- {时间} 项目启动
- {时间} 创建开发计划：doc/plan.md

## 任务 1：{标题}
- {时间} 创建开发子智能体（ID: {id}），交办任务
- {时间} 开发完成，输出文件：{路径}
- {时间} 创建测试子智能体（ID: {id}），交办测试任务
- {时间} 测试结果：PASS/FAIL
  - FAIL → 反馈内容摘要：{...}
- {时间} Resume 开发子智能体（ID: {id}），传递反馈
- {时间} 开发修改完成
- {时间} Resume 测试子智能体（ID: {id}），重新审查
- {时间} 测试结果：PASS → 任务 1 完成

## 任务 2：{标题}
- ...
```

### 8.3 子智能体日志格式

```markdown
# {角色}日志 - {任务标识}

- {时间}，收到任务：{任务描述}
- {时间}，读取文件：{文件列表}
- {时间}，开始 {开发/测试}
- {时间}，{关键操作}：{描述}
- {时间}，完成，输出：{结果摘要}
```

## 九、常见问题与踩坑指南

### 9.1 技术类问题

| 问题 | 原因 | 解法 |
|------|------|------|
| Agent(resume) 找不到 ID | 路径不对，有双层子目录 | `ls -lt ~/.claude/projects/*/*/subagents/agent-*.meta.json \| head -3` |
| Resume 报 "Agent type not found" | 未指定 subagent_type | 必须指定与创建时一致的 subagent_type |
| 子智能体回复不在 UI 显示 | 回复在 JSONL 文件中 | 读 JSONL 文件查看 |
| SendMessage 只写 inbox 不唤醒 | 旧版行为 | 使用 Agent(resume) 方式 |
| JSONL 文件膨胀 | 每次resume重写历史 | 不影响上下文窗口，只是磁盘问题 |

### 9.2 流程类问题

| 问题 | 原因 | 解法 |
|------|------|------|
| 开发智能体不理解测试反馈 | 传给新智能体而非 resume | 修正循环必须 resume 原智能体 |
| 测试智能体误判 | 检查标准不够明确 | 增加验证步骤（如"用答案乘回来验证"） |
| 经验库太长，子智能体读不完 | 积累了太多条目 | 定期精简，只保留通用的、高价值的 |
| 开发智能体重复踩坑 | 没读经验库 | 在 prompt 中强制要求先读 lessons-learned.md |

## 十、适用场景与不适用场景

### 适用场景

- 大型开发任务，可拆分为独立子任务
- 需要反复迭代验证的任务（如 UI 开发、动画制作）
- 需要严格质量把控的项目
- 团队协作模拟（开发 + 审查分工）

### 不适用场景

- 简单的一次性任务（一个子智能体就够）
- 任务间强耦合、无法独立验证
- 不需要迭代修正的场景

## 十一、与其他方法的对比

| 维度 | 单智能体长对话 | 双窗口手动切换 | 本方法论（主智能体编排） |
|------|---------------|---------------|----------------------|
| 上下文管理 | 差（窗口爆满） | 中（手动管理） | 好（自动隔离+传递） |
| 迭代效率 | 低（全在对话中） | 中（手动传递） | 高（自动循环） |
| 可追溯性 | 差（对话太长） | 中（两份记录） | 好（三层日志） |
| 经验积累 | 差（对话结束即丢失） | 差（依赖人工总结） | 好（自动写入经验库） |
| 实施复杂度 | 低 | 中 | 高（需要精心设计模板） |

## 十二、快速启动清单

```

     └── logs/                     # 日志目录

     .claude/agents/dev-agent.md
     .claude/agents/tester-agent.md

□ 3. 主智能体编写开发计划
     → 拆分为可独立验证的子任务

□ 4. Phase 0：主智能体搭建框架

□ 5. 逐任务执行开发-测试循环

□ 6. 整体集成测试

□ 7. 总结经验，更新经验库
```
